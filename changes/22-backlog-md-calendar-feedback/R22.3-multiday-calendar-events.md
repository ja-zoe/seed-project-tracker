# R22.3 — Multi-day events render on every day they span

**Status:** done
**Files:** `src/components/semester-calendar.tsx`

## Spec

**Problem:** The month grid keys events to their start day only — `semester-calendar.tsx:463`:
`events.filter((e) => isSameDay(new Date(e.startsAt), day))`. A meeting/event with an `endsAt` on a later
day (e.g. a weekend retreat, a multi-day conference) appears only on its first day; the remaining days look
free, which misleads scheduling.

**Approach:**

- **Month grid:** replace the `isSameDay(startsAt, day)` filter with a span check: the event appears on
  `day` when `startOfDay(startsAt) <= day <= endOfDay(endsAt ?? startsAt)`. Add a small
  `eventCoversDay(event, day)` helper next to `isSameDay` (date-only comparison, consistent with the
  existing `isSameDay` convention; `endsAt: null` degrades to exactly today's single-day behavior).
- **Continuation rendering:** on days after the first, render the same event chip but with the start-time
  prefix replaced by a `(cont.)` marker (or simply title-only) so a spanning event doesn't read as starting
  fresh each morning. Keep it one chip style — no cross-cell spanning bars (that's a layout rewrite the
  Forest Floor calendar doesn't need).
- **Day-click / details modal:** the day detail listing (which reuses the same per-day filter) picks up the
  fix automatically if the filter is the shared helper — verify it does rather than having its own copy.
- **Agenda/week list (line ~570):** groups events into the week of `startsAt` — leave as-is; an agenda is a
  list of events (each shown once, already displaying its start–end range), not a day grid. Out of scope.
- **"Next meeting" picker (line ~650):** keyed on `startsAt` distance — unaffected, leave as-is.
- No state, data-flow, or permission changes — pure render logic in one client component.

**Non-obvious constraints:** the calendar builds its `days` array with `null` padding cells
(`semester-calendar.tsx:455-460`) — the covers-day helper must be applied only to real `Date` cells. Watch
the existing local-time vs `toISOString()` mixing in this file (line 127 vs 483-488, and the R14 date-only
UTC work): do the day comparison in **local time** like `isSameDay` does, not via ISO string slicing.
