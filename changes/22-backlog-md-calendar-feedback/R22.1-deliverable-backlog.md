# R22.1 — Deliverable backlog

## Spec

**Problem:** A lead who over-scoped their semester has no way to defer a deliverable without deleting it
(losing the record and its subtasks) or leaving it overdue (skewing the timeline and eventually tripping
red-flag Condition A → project BEHIND). There is no backlog concept anywhere in the schema or app.

**Approach:**

- **Schema:** `backlog Boolean @default(false)` on `Deliverable` (see `_set.md` DB changes for the SQL and
  the apply-schema.ts procedure). `targetDate` stays required and untouched, so moving an item back from
  backlog restores it exactly. State lives in the DB; no client/local state beyond the optimistic UI.
- **Server action:** `setDeliverableBacklog(deliverableId, backlog: boolean)` in `deliverables.ts`, gated
  exactly like `updateDeliverable` (MANAGE_MILESTONES or project LEAD/SUBLEAD via `getProjectMembership`).
  `revalidatePath` the project page **and** `/timeline` (cross-surface freshness — CONTEXT.md parity rules).
- **Project page / SortableDeliverables:** active deliverables render as today; backlogged ones move to a
  collapsed **"Backlog (N)"** section below the active list (always visible when N > 0 — PM transparency,
  see `_set.md` open question 2). Each row gets a "Move to backlog" / "Restore" control (Phosphor
  `Archive` / `ArrowCounterClockwise`, Bold, `.clickable-icon`), shown when `canManage`. Backlogged rows are
  excluded from drag-reorder and render muted (reduced opacity, no overdue red — deferred work isn't late).
  Data flow: control → `setDeliverableBacklog` → revalidate → row moves sections.
- **DeliverableModal:** a Backlog toggle in the edit dialog for parity with the inline control.
- **Timeline page:** exclude `backlog: true` from the Gantt query — deferred work doesn't belong on the
  semester timeline.
- **Red-flag (`red-flag.ts`):** Condition A ignores backlogged deliverables
  (`some((d) => !d.backlog && d.targetDate < cutoff)` or filter in the query). See open question 2.
- **Excel export:** backlogged deliverables export under a separate "Backlog" grouping (or a Backlog
  column), not silently dropped.
- **MCP:** `list_deliverables`/get tools return `backlog`; `update_deliverable` accepts an optional
  `backlog` boolean (same gate as the web action). Per CONTEXT.md, the MCP tool duplicates the gate — keep
  it in sync with the server action.

**Non-obvious constraints:** the parity surface set for deliverables is five-wide (create page, modal,
inline controls, server actions, MCP) — this feature intentionally omits the create page (nobody creates a
deliverable directly into backlog; confirm if that's wanted). `createDeliverable` defaults `backlog=false`
implicitly via the column default.
