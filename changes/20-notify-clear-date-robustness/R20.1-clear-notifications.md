# R20.1 — Clear notifications

**Status:** planned — confirm Open Question 1 (delete vs soft-hide) in `_set.md`.
**Files:** `src/components/notification-bell.tsx` (UI), a new clear endpoint
`src/app/api/notifications/clear/route.ts` (or extend `notifications/read/route.ts` with a DELETE), and
possibly `src/app/api/notifications/route.ts` (unchanged GET).

## Spec

**Problem:** The notification bell (`notification-bell.tsx`) can **mark** notifications read (per-item and
"mark all read" → `POST /api/notifications/read`), but there is **no way to clear/remove them** — read
notifications pile up in the dropdown with no dismiss. Users want to clear the list.

**Approach (decided: hard-delete — confirm Open Question 1):**

1. **Endpoint.** Add `src/app/api/notifications/clear/route.ts`:
   - `POST` with `{ id }` → delete that one notification (`prisma.notification.deleteMany({ where: { id, userId } })`).
   - `POST` with empty body → delete all the user's notifications (`deleteMany({ where: { userId } })`).
   Auth via `auth()` → 401 if no session (mirror the existing `read/route.ts` shape). Scope every delete to
   `userId: session.user.id` so a user can only clear their own. (Alternatively a `DELETE` method — match the
   project's existing POST-style for consistency with `read`.)

2. **UI.** In `notification-bell.tsx`:
   - Add a **"Clear all"** button in the dropdown header (next to "Mark all read"), shown when the list is
     non-empty. On click → call the clear endpoint (no body), then `setNotifications([])` + `setUnreadCount(0)`.
   - Add a per-item **dismiss (×)** control on each notification row → call clear with `{ id }`, then remove
     that item from state and decrement `unreadCount` if it was unread.
   - Empty state: when no notifications, show the existing/"No notifications" message.
   - Style the new controls with the R19 affordance convention (`.clickable` / `.clickable-icon`; the
     dismiss × is fine as `.clickable-icon`, "Clear all" can be `.clickable-danger` since it's destructive).

**State:** notification list/count already live in `notification-bell.tsx` (`useState`, polled every 60s).
Clearing mutates that local state optimistically after the fetch resolves. **Data flow:** click → fetch
clear endpoint → on ok, update local state. No revalidatePath (client-fetched, not server-rendered).

**Non-obvious constraints:** every delete must be scoped to the session user. "Clear all" deletes read AND
unread (it's a clear, not a mark-read). Keep the 60s poll working afterward (a cleared list stays empty
until new notifications arrive).
