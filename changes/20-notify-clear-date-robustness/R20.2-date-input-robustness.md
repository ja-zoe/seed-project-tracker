# R20.2 — Date-input robustness (invalid dates must not crash the page)

**Status:** planned — confirm Open Question 2 (shared guard) in `_set.md`.
**Files:** the date-accepting server actions (`src/lib/actions/action-items.ts`, `deliverables.ts`,
`projects.ts`, `status-updates.ts`), a new shared helper (e.g. `src/lib/date.ts` `parseDateInput`), and the
date `<input type="date">` sites: `project-modal.tsx`, `deliverable-modal.tsx`, `subtask-modal.tsx`,
`action-item-modal.tsx`, `action-items-section.tsx` (inline deadline), `sortable-deliverables.tsx` (inline
dates), `projects/[id]/deliverables/new/page.tsx`, `.../subtasks/[sid]/edit/page.tsx`.

## Spec

**Problem:** Entering an impossible date such as **06/31/2026** crashes the **whole page** (runtime error)
instead of being caught in the date field. Dates are entered via raw native `<input type="date">` and the
server actions parse them with unguarded `new Date(raw)` (e.g. `action-items.ts`:
`const deadline = deadlineRaw ? new Date(deadlineRaw) : null`). An invalid/out-of-range value yields an
`Invalid Date`, which then throws when Prisma tries to persist it (or when a later `.toISOString()` /
`toLocaleDateString` runs on it), bubbling up as an unhandled error that takes down the page.

**Approach:** Make invalid dates a *handled, field-level* condition end to end.

1. **Root-cause first.** Reproduce 06/31/2026 in a date field (e.g. an action-item deadline or subtask due
   date) and capture the exact throw site (parse vs persist vs render). Fix that path specifically, then
   apply the general guards below so no date path can crash.

2. **Shared safe parse (server).** Add `src/lib/date.ts`:
   ```ts
   /** Returns a valid Date, null for empty, or throws a typed ValidationError for an invalid value. */
   export function parseDateInput(raw: FormDataEntryValue | null): Date | null
   ```
   It trims, returns `null` for empty, builds the Date, and treats `isNaN(d.getTime())` (and impossible
   calendar dates) as invalid. Replace every unguarded `new Date(raw)` in the date-accepting actions with
   this. On invalid, the action must **not throw an unhandled error** — return a structured error the caller
   can show (or no-op the field), so a bad value never reaches Prisma or `.toISOString()`.

3. **Field-level validation (client).** The user must see the error "in the date picker." Add inline
   validation to the date inputs: on change/blur, if the value is an impossible date, show a small inline
   error message near the field (Forest Floor clay `#A4503C`) and block submit, rather than posting it.
   Because there are ~8 native date inputs, prefer a small shared client wrapper (e.g. `DateField`) that
   renders `<input type="date">` + an inline error slot and runs the same validity check, so the behavior is
   consistent everywhere. (If a full wrapper is too invasive for this fix, at minimum guard the submit
   handlers in the modals/inline editors to validate before calling the server action.)

4. **Render guards.** Anywhere a stored date is formatted for display
   (`new Date(iso).toLocaleDateString(...)` / `.toISOString()` in `sortable-deliverables.tsx`,
   `action-items-section.tsx`), tolerate a bad/missing value (guard `isNaN`) so display never throws.

**State:** validation state is local to each date field/modal (client); the server helper is pure. **Data
flow:** input change → client validity check → (valid) server action → `parseDateInput` → Prisma; (invalid)
inline error, no submit.

**Non-obvious constraints:** native `<input type="date">` behavior varies by browser/locale — don't rely on
it to reject impossible dates; validate explicitly. Keep the existing `timeZone: "UTC"` formatting (the app
deliberately formats dates in UTC — see R14 date-UTC work) so this fix doesn't shift displayed dates.
