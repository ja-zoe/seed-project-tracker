# R12.2 — Multiple pending project standings + configurable late window

**Status:** planned
**Files:** `prisma/schema.prisma`, `src/lib/lead-meeting.ts`, `src/lib/actions/status-updates.ts`,
`src/lib/actions/settings.ts`, `src/app/(app)/pm/settings/page.tsx`,
`src/app/(app)/projects/[id]/status/new/page.tsx`, `src/components/status-submit-switcher.tsx` (new),
`src/app/(app)/projects/[id]/page.tsx`, `src/app/(app)/dashboard/page.tsx` (+ regenerated client)

## Spec

**Problem (Defect B):** `getActiveLeadMeeting(projectId)` returns a single meeting — the *latest*
in-window one (`findFirst orderBy startsAt desc`). When several lead meetings are simultaneously within
their submission windows, only that one gets a "Submit Project Standing" button; after it's submitted the
earlier meeting(s) never offer one. Reproduced: Fall 2026 had three in-window lead meetings (Jun 29,
Jun 30, Jun 30) but only one was returned, and `getStatusSubmissionState` then reported `canSubmit=false`
once it was submitted. Also, the current window has **no lower bound** (`startsAt <= now+window` is always
true for past meetings), so a pending-set model would resurrect ancient un-submitted meetings forever.

**Approach:**

*DB (configurable late window).* Add `Settings.statusLateWindowDays Int @default(3)` — how many days
**after** a lead meeting starts a status update may still be submitted (late) before the meeting is
considered missed and drops off the pending list. Mirrors `statusSubmitWindowDays`.
- DDL (column add, can share a txn): `ALTER TABLE "Settings" ADD COLUMN IF NOT EXISTS "statusLateWindowDays" INTEGER NOT NULL DEFAULT 3;`
  via `scripts/apply-schema.ts`; add the field to `schema.prisma`; `prisma generate`.
- `settings.ts updateSettings`: parse `statusLateWindowDays` (positive int, fallback 3) and persist on both
  the `update` and `create` branches of the singleton upsert.
- `pm/settings/page.tsx`: add a number input (`name="statusLateWindowDays"`, `min=0 max=30`,
  `data-testid="status-late-window"`) next to the submit-window field, labelled "Late submission window
  (days after a lead meeting)".

*Pending-set model.* In `src/lib/lead-meeting.ts`:
- Add `getPendingLeadMeetings(projectId): Promise<{ meeting: { id; title; startsAt }; isLate: boolean }[]>`.
  A lead meeting is **pending** when its window is open and it's unsubmitted for this project:
  `startsAt ∈ [now - statusLateWindowDays, now + statusSubmitWindowDays]` AND
  `statusUpdates: { none: { projectId } }` (uses the `CalendarEvent.statusUpdates` back-relation), filtered
  to `type: "LEAD_MEETING"`, `semesters: { has: project.semester }`, ordered `startsAt asc`. `isLate = now > startsAt`.
- Reframe `getStatusSubmissionState(projectId)` to `{ pending, count, canSubmit }` where
  `pending = getPendingLeadMeetings(...)`, `count = pending.length`, `canSubmit = count > 0`.
- Remove `getActiveLeadMeeting` (its two callers move to the pending model).

*Submit action.* `status-updates.ts submitStatusUpdate(projectId, formData)`:
- Read `calendarEventId` from the form (hidden field). Validate it's in `getPendingLeadMeetings(projectId)`
  (membership already checked: lead/sublead). If not pending → throw. Derive `meetingDate` + `isLate` from
  that meeting. Create the update linked to it (unchanged otherwise).
- After create, `redirect` back to `/projects/${projectId}/status/new` so any remaining pending meetings
  surface for immediate submission; the page itself redirects to the project when none remain.

*Submit page + switcher.* `status/new/page.tsx`:
- Load `pending = getPendingLeadMeetings(id)`; if empty → `redirect(/projects/${id})`.
- Render a new client component `StatusSubmitSwitcher` (`src/components/status-submit-switcher.tsx`) given
  `projectId` + the `pending` list (id, title, startsAt ISO, isLate) + the bound submit server action.
  It holds `selectedIndex`, shows a header "You have N Project Standing Updates to submit" (N = count),
  prev/next controls labelled with each meeting's date, the current meeting's date + a "marked late" notice
  when `isLate`, and the existing form fields. A hidden `<input name="calendarEventId">` carries the
  selected meeting id. Reuse the existing field markup + `SubmitButton`. (Per-meeting draft text is not
  preserved across switches — you fill and submit one at a time; on submit it leaves the pending list and
  the page reloads to the remaining ones.)

*Project page.* Replace `submissionState.canSubmit` usage with the new shape: `canSubmitStatus = isLeadHere
&& state.canSubmit`; the button label is "Submit Project Standing" when `state.count === 1`, else
`You have ${state.count} Project Standing Updates to submit`. Links to `/projects/${id}/status/new`.

*Dashboard.* `dashboard/page.tsx` uses `getStatusSubmissionState(p.id)`; update to the new shape
(`submissionState.canSubmit` still exists, so the per-card `canSubmit` line keeps working).
