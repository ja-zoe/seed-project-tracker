# R12.1 — Calendar opens on the correct semester (and month)

**Status:** planned
**Files:** `src/app/(app)/calendar/page.tsx`, `src/components/semester-calendar.tsx`

## Spec

**Problem (Defect A):** The calendar's default semester is wrong. `calendar/page.tsx` builds
`allSemesters = [...new Set(...)].sort().reverse()` and uses `activeSemester = semesterParam ?? allSemesters[0]`.
`.sort()` is **lexicographic**, so for `["Fall 2026","Test 2026"]` it returns `["Fall 2026","Test 2026"]`
and `.reverse()` makes `allSemesters[0] = "Test 2026"`. Every project is in `"Fall 2026"`, so the calendar
opens on a semester where the user's Fall-2026 lead meetings aren't pinned — they're invisible until the
user manually switches the dropdown — even though the project's "Submit Project Standing" button (keyed
off the *project's* semester via `getActiveLeadMeeting`) finds them. Reproduced live:
`allSemesters.sort().reverse()[0]` → `"Test 2026"` while projects are all `"Fall 2026"`.

Semesters are free-text strings with no intrinsic dates, so any string sort is meaningless. The default
should track **time**: open on the semester whose activity is happening *now*.

**Approach:**
- In `calendar/page.tsx`, when no `?semester=` param is given, pick the default semester as the one whose
  events are **nearest to today**. Implementation: fetch all calendar events' `{ semester, semesters, startsAt }`
  (one `findMany`), and for each known semester compute `minDist = min(|startsAt - now|)` over every event
  that belongs to it (`event.semester === s` OR `s ∈ event.semesters`). Default = the semester with the
  smallest `minDist`. Semesters with no events (project-only) get `Infinity` and sort last but remain
  selectable. If there are no events at all, fall back to the existing project-derived list (any order).
- Keep `allSemesters` as the **dropdown list**, but order it by the same nearest-today distance (soonest
  activity first) so the picker is sensible; an explicit `?semester=` param still wins.
- `semester-calendar.tsx`: the month view defaults `displayMonth/displayYear` to `events[0]` (the
  *earliest* event of the active semester). Change it to default to the event **nearest today** within the
  active semester (fall back to `today` when the semester has no events), so a multi-month semester opens
  on the current month rather than its first month.

**Non-goals:** no schema change; no change to how events are pinned or queried (`semesters: { has }`
stays). Visibility filtering (`VIEW_LEAD_MEETINGS`) is untouched.
