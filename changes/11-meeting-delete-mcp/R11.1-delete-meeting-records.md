## Spec

**Problem:** Meeting records can be created (`createMeetingRecord`, gated `POST_MEETING_TRACKING`) but
never deleted. A duplicate or mistaken record is permanent — it skews the project-standing history and
feeds `runRedFlagDetection` (which reads meeting `status` + `goalMet` to escalate a project to BEHIND),
so a bad record can wrongly hold a project in BEHIND. There is no affordance to remove one.

**Approach:**

*Permission (DB change).* Add a new `Permission` value `MANAGE_MEETING_RECORDS`. Deletion of records is
a heavier, PM-level action, kept distinct from the create permission per the user's decision.
- Apply DDL statement-by-statement via `scripts/apply-schema.ts` (ALTER TYPE ADD VALUE can't share a txn):
  `ALTER TYPE "Permission" ADD VALUE IF NOT EXISTS 'MANAGE_MEETING_RECORDS';`
- Add `MANAGE_MEETING_RECORDS` to the `Permission` enum in `prisma/schema.prisma`; `pnpm prisma generate`.
- `prisma/seed.ts`: append `Permission.MANAGE_MEETING_RECORDS` to `PM_PERMISSIONS` only (matches
  `POST_MEETING_TRACKING`, which is PM-only). Run `pnpm db:seed`.

*Server action.* Add `deleteMeetingRecord(meetingRecordId: string)` to `src/lib/actions/meeting-records.ts`:
- `await requirePermission(Permission.MANAGE_MEETING_RECORDS)`.
- `findUnique` the record (`select: { id, projectId }`); if missing, `return` (no-op) — the UI revalidates.
- `prisma.meetingRecord.delete({ where: { id } })`. **Action items are preserved:** `ActionItem.meetingId`
  is an optional FK (Prisma default `onDelete: SetNull`), so items carried at that meeting survive with a
  null `meetingId` — verify this holds; do **not** cascade-delete action items.
- Re-run `runRedFlagDetection(projectId)` afterward (history changed — a record that forced BEHIND may be
  gone; let auto-detection re-evaluate, respecting `statusOverride`).
- `revalidatePath` for `/projects/${projectId}`, `/projects/${projectId}/history`, and `/projects`.

*UI.* New client component `src/components/meeting-record-controls.tsx` exporting
`MeetingRecordControls({ recordId }: { recordId: string })`, mirroring `StatusUpdateControls`' delete
microinteraction exactly: `confirmingDelete` boolean state, `useTransition`, the shared `InlineConfirm`
from `sortable-deliverables`, mono "Delete" → "Delete?" + confirm/cancel, hover `#A4503C`. Delete only —
no edit (not requested). On confirm: `startTransition(() => deleteMeetingRecord(recordId))`.
- **Project page** (`projects/[id]/page.tsx`): compute
  `const canManageMeetingRecords = permissions.includes(Permission.MANAGE_MEETING_RECORDS);` and, in the
  Meeting Records section (~line 596, the date/badge header row), render
  `{canManageMeetingRecords && <MeetingRecordControls recordId={record.id} />}`.
- **History page** (`projects/[id]/history/page.tsx`): compute the same boolean (page already loads
  `permissions`), and in the meeting-card header row (~line 199) render the control when allowed.
- Both pages are server components; the control is a client island gated by the server-computed boolean,
  so members/leads without the permission never receive it.
