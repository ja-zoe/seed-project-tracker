# R11.2 — MCP capability expansion

**Status:** tests passing
**Files:** `src/app/api/mcp/route.ts` (depends on R11.1's `MANAGE_MEETING_RECORDS` in the generated client)

## Spec

**Problem:** Going through the MCP server's capabilities (`/api/mcp`), three gaps stand out:
1. **Meeting records are entirely absent** — there is no way to read, create, or delete the
   `MeetingRecord` domain via MCP, even though it drives project status + red-flag detection.
2. **`list_calendar_events` leaks restricted meetings.** Its `type` enum predates Set 10
   (`PROJECT_MEETING`, `NON_PROJECT_EVENT` only) and it applies **no** `VIEW_LEAD_MEETINGS` filter — so a
   member's MCP token can read `LEAD_MEETING`/`EBOARD_MEETING` events, contradicting the visibility rule
   enforced in `calendar/page.tsx` and `api/calendar/ics/route.ts`.
3. **Status updates are write-only** — `create_status_update` exists but there's no read tool.

**Approach:** all changes are in `src/app/api/mcp/route.ts`. Bump `serverInfo.version` `2.2.0` → `2.3.0`.

*New tools (TOOLS array + `executeTool` cases):*

- `list_meeting_records` — read. Params: `projectId` (required), `limit?` (default 20, max 50). Access
  gate identical to `get_project_detail`: caller needs project membership **or** `VIEW_ALL_PROJECTS`,
  else `{ error: "No access to this project" }`. Returns records newest-first:
  `{ id, meetingDate, status, goalMet, keyBlockers, notes, recordedByName }`.

- `create_meeting_record` — write, gated `POST_MEETING_TRACKING` (mirrors the `createMeetingRecord` server
  action). Params: `projectId`, `meetingDate` (ISO), `status` (enum `ON_TRACK|AT_RISK|BEHIND`), `goalMet?`
  (boolean), `keyBlockers?`, `notes?`. Replicate the action's side effects so MCP and UI stay consistent:
  create the record (`recordedById = user.id`); if the project's `statusOverride` is false, set
  `project.status = status`; `await carryOverActionItems(projectId)`; `await runRedFlagDetection(projectId)`.
  Import `carryOverActionItems` from `@/lib/actions/action-items` and `runRedFlagDetection` from
  `@/lib/red-flag`. Returns `{ created: { id, meetingDate, status } }`.

- `delete_meeting_record` — write, gated **`MANAGE_MEETING_RECORDS`** (the R11.1 permission). Params:
  `meetingRecordId`. Look up the record (`select { id, projectId }`); if missing
  `{ error: "Meeting record not found" }`. Delete; then `await runRedFlagDetection(projectId)`. Returns
  `{ deleted: { id } }`. (Action items survive via the SetNull FK, same as R11.1.)

- `list_status_updates` — read. Params: `projectId?`, `status?` is N/A; `limit?` (default 20, max 50).
  Scope mirrors `list_action_items`: if `projectId` given, require membership or `VIEW_ALL_PROJECTS`;
  if omitted and caller lacks `VIEW_ALL_PROJECTS`, restrict to projects the caller is assigned to.
  Returns newest-first: `{ id, projectId, projectName, meetingDate, plannedWork, actualProgress, blockers,
  nextWeekGoals, needsHelp, helpNeeded, isLate, submittedByName }`.

*Calendar visibility fix:*

- `list_calendar_events`: extend the `type` enum to
  `["PROJECT_MEETING","NON_PROJECT_EVENT","LEAD_MEETING","EBOARD_MEETING"]`. In the handler compute
  `canSeeRestricted = permissions.includes(VIEW_LEAD_MEETINGS)`. When the caller **cannot** see restricted
  meetings, add `type: { notIn: ["LEAD_MEETING","EBOARD_MEETING"] }` to the `where`; if they explicitly
  pass a restricted `type` they aren't allowed to see, return `{ events: [] }` (don't honor the filter).
  Mirrors `calendar/page.tsx`.
- `create_calendar_event` / `update_calendar_event`: add `LEAD_MEETING`/`EBOARD_MEETING` to their `type`
  enums (already gated by `MANAGE_CALENDAR`). Add an optional `semesters: string[]` param; lead/eboard
  meetings are matched by `CalendarEvent.semesters` (see `lead-meeting.ts`), so when the type is
  `LEAD_MEETING`/`EBOARD_MEETING` and no `semesters` is supplied, default it to `[semester]` so the
  meeting actually governs that semester's projects. For other types `semesters` stays as given (or `[]`).
