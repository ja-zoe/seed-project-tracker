# R19.2 — Site-wide rollout + audit (with Playwright verification)

**Status:** planned — depends on R19.1's utility classes. Confirm Open Question 2 (scope) in `_set.md`.
**Files:** broad — components under `src/components/` and interactive elements across `src/app/(app)/**`.
See the audit matrix below; treat it as the working checklist (update it as the audit proceeds).

## Spec

**Problem:** Interactivity is signalled inconsistently across the app. R19.1 defines the cue; this feature
**applies it everywhere** so that on hover every clickable element gives a clear visual cue, clickable icons
always switch to a pointer cursor + color, and similarly-styled interactive objects behave identically
(deliverables, inline-editable items, subtasks, etc.).

**Approach:** Audit the entire site for interactive elements and apply the R19.1 classes consistently,
replacing ad-hoc `hover:` treatments where they diverge. Apply the **row-vs-control distinction**:
container rows that open get `.clickable-row` (subtle tint, **default cursor**); the controls inside and all
other clickable icons/links/buttons get `.clickable-icon` / `.clickable` (pointer + color), destructive →
`.clickable-danger`. Static/display elements are left alone (Open Question 2). Work component-by-component
using the matrix below.

**Audit matrix (verify + fix each; check off during implementation):**
- [ ] `components/sortable-deliverables.tsx` — deliverable rows + subtask rows → `.clickable-row` (tint, no
      pointer, helps tell rows apart); inline-edit triggers, `InlineConfirm` buttons, drag handles, the
      StatusPill → `.clickable-icon` (pointer + color).
- [ ] `components/action-items-section.tsx` — row inline-edit triggers (description/owner/deadline), close/
      reopen/edit/delete controls, "add" button.
- [ ] `components/action-item-modal.tsx` — footer buttons, any field affordances.
- [ ] `components/meeting-record-controls.tsx` — delete trigger (→ `.clickable-danger`).
- [ ] `components/status-update-controls.tsx` — edit/delete controls.
- [ ] `components/project-modal.tsx`, `components/projects-list.tsx` — row links/cards, action buttons.
- [ ] `components/semester-calendar.tsx` — clickable day cells / events.
- [ ] `app/(app)/account/mcp-token-section.tsx` — copy/regenerate/revoke icon buttons (`.clickable-icon`).
- [ ] `app/(app)/pm/users/page.tsx` — approve/suspend/reactivate/delete (R18.2) controls, role controls.
- [ ] App shell sidebar/nav (`app/(app)/layout.tsx` + nav components) — nav links, sign-out.
- [ ] Sweep the rest of `src/components/` and `src/app/(app)/**` for any `onClick`/`<button>`/`<a>`/
      `role="button"` without a hover cue, and any icon button without `cursor-pointer`.

**State:** none (presentational only). **Constraints:** no behavior changes — only affordances; preserve
existing focus rings, `data-testid`s, and `aria` attributes; keep within Forest Floor tokens; no new deps.

## Tests

Playwright-driven (the request requires it). Add/extend an e2e under the existing test setup; drive the
deployed-or-local app and assert the affordance on representative elements of each audited surface:

- [ ] `pnpm build` clean.
- [ ] **Pointer cursor (controls):** for a sample clickable icon/button in each surface (account copy
      button, a deliverable inline-edit pencil, an action-item delete), assert
      `getComputedStyle(el).cursor === "pointer"`.
- [ ] **Default cursor (rows):** a deliverable/subtask container *row* has `cursor` **not** `pointer`, while
      a control inside that same row does — confirming the row-vs-control split.
- [ ] **Row hover tint:** capture a row's `background-color` at rest, dispatch hover, assert it changes
      (subtle tint) so adjacent rows are distinguishable.
- [ ] **Hover color cue (controls):** capture a control's `color` at rest, hover, assert the computed
      `color` changes (forest `#2E4034`, or clay `#A4503C` for destructive).
- [ ] **Consistency:** the same class of object (all subtask rows; all inline-edit triggers) exhibits
      identical behavior — assert across several instances, not just one.
- [ ] **No false affordance:** a representative static display element (a non-button status badge) gets
      neither pointer nor hover cue (guards Open Question 2).
- [ ] Spot-check screenshots of a couple of key surfaces (deliverables, action items) for the reviewer.
