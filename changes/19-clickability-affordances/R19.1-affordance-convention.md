## Spec

**Problem:** There's no shared, named convention for "this element is clickable." Hover treatments are
ad-hoc (`hover:text-foreground` here, a tooltip there, a cursor change somewhere else), so the rollout in
R19.2 would otherwise reinvent the cue per component and drift. Define the cue once.

**Decided distinction (user 2026-06-30): a clickable *container row* and the *controls inside it* get
different cues.** A deliverable/subtask row that can open should give a subtle hover cue (to show hover and
to make it easy to tell *which* row you're on) but keep the **default cursor** — the pointer is reserved for
the actionable controls inside the row (inline-edit icons, buttons), so the pointer always points at the
thing that acts. This avoids "the whole row is a pointer" ambiguity and improves row differentiation.

**Approach:** Add small utility classes in `globals.css` (Tailwind v4 — `@layer utilities` or plain CSS
classes bridged via the existing `@theme inline` tokens; no config file). Built on Forest Floor tokens, no
new colors, no underlines/shadows/gradients (CONTEXT style rules).

- **`.clickable-row`** — interactive *container rows* (deliverable/subtask rows that open): a subtle hover
  cue — a faint row tint (e.g. `background` shifts to a very low-opacity foreground/muted wash) and/or a
  gentle text-color lift — with **`cursor: default`** (no pointer). Purpose: hover feedback + "which row am
  I on" differentiation. `transition-colors`, ~150ms.
- **`.clickable-icon`** — icon buttons / inline controls (including those inside a `.clickable-row`):
  **`cursor: pointer;`** + a `color` shift on hover (muted-foreground → foreground `#2E4034`), matching the
  icon-button idiom already on the account page.
- **`.clickable`** — interactive text/labels/links that are not container rows: `cursor: pointer;` + color
  shift to foreground on hover.
- **`.clickable-danger`** — variant of the pointer cues whose hover color is clay `#A4503C` (delete/remove
  triggers), matching the existing destructive convention.

Keep them composable with existing Tailwind utilities (a component can add `className="clickable-icon"` or
layer `hover:` overrides). Document the rule in CONTEXT.md "Standing decisions": *every interactive element
carries a hover cue; clickable **icons/controls** get pointer cursor + color shift
(`.clickable`/`.clickable-icon`, danger → `.clickable-danger`); clickable **container rows** get a subtle
hover tint with default cursor (`.clickable-row`) since the pointer belongs to the controls inside; never
signal interactivity on static display elements.*

**State:** none — pure CSS + docs. **Constraints:** respect `prefers-reduced-motion` (the color transition
is fine; avoid motion-heavy cues). Keep the row tint subtle enough that it reads as *hover*, not *selected*.
Don't remove existing keyboard focus rings.
