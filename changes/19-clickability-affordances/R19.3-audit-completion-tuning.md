## Spec

**Problem:** R19.2 claimed a site-wide audit but it was incomplete and one value is mistuned. Three concrete
defects from user review:
1. **Users & Roles page (`/pm/users`) has no hover cues** — its controls (approve/activate selects + buttons,
   role permission toggles, role `<details>`/`<summary>`, "Save role", "Delete role", create-role form
   buttons, etc.) were marked `[x]` in R19.2's matrix but do not actually show interactivity on hover.
2. **Inline status pill is not obviously clickable** — the idle StatusPill label button in
   `sortable-deliverables.tsx` uses `hover:opacity-80 transition-opacity` but **no `cursor-pointer`**, so it
   doesn't signal it's clickable (it opens the status dropdown).
3. **Deliverable row hover tint too strong** — `.clickable-row:hover` uses
   `color-mix(in srgb, var(--foreground) 4%, transparent)`; the user wants it more subtle.

The deeper requirement: **EVERY clickable element in EVERY view must carry the cue** (per the set's plan),
not just the surfaces R19.2 happened to touch.

**Approach:**

1. **Tint (item 3).** In `globals.css`, reduce `.clickable-row:hover` background to ~2% (e.g.
   `color-mix(in srgb, var(--foreground) 2%, transparent)`) — subtle enough to read as hover, not selection.
   Confirm visually + update the R19.1 note. (One-line change.)

2. **Status pill (item 4) — a status pill IS a button (deliverable AND subtask).** The `StatusPill`
   component in `sortable-deliverables.tsx` is shared by **both** the deliverable header status and the
   per-subtask status (each opens a dropdown to change status; subtask confirm → `updateSubtaskStatus`). Add
   `cursor-pointer` to its idle label button (the branch currently `hover:opacity-80 transition-opacity`;
   keep `cursor-default` on the confirming/inert branch). Because the component is shared, this one fix makes
   **both** deliverable and subtask status read as clickable. Keep the existing opacity cue.
   **Rule (user clarification 2026-06-30):** a status pill that triggers a status-change dropdown is
   **interactive** → it gets the affordance. This includes the **subtask status** (always changeable) and
   the **deliverable status** (changeable *unless locked*). The **only** static status case is the **locked
   deliverable status** (`deliverable-locked-badge`, "Status is locked because…") — it can't be changed, so
   no cue. The decorative `status-bullet` dot beside the pill is `aria-hidden` visual-only — it stays without
   a pointer. Note: `ProjectStatusBadge`/`TimelineStatusBadge` spans in `status-badge.tsx` (project header,
   projects list, history, meeting records) are currently **display-only** (not wired to a change dropdown),
   so they stay static *as coded* — see the Open Question about project status.

3. **Users & Roles page (item 1).** Apply the R19.1 convention to **all** interactive elements in
   `pm/users/page.tsx`: every `<button>`/submit (approve, activate, suspend, reactivate, delete-user,
   save-role, create-role) → `.clickable`/`cursor-pointer` (+ `.clickable-danger` for destructive); the role
   permission checkboxes/labels and the `<details><summary>` disclosure rows → pointer + hover cue; any
   `<select>` controls → `cursor-pointer`. Don't signal static text/badges.

4. **Exhaustive re-audit (the real fix for item "every view").** Walk **every** route under
   `src/app/(app)/**` (dashboard, projects + all sub-routes: deliverables/timeline/status/meeting/members/
   action-items/history, my-tasks, action-items, account, pm/users, pm/review, pm/settings, profile/setup,
   pending) and every component they render. For each interactive element (`<button>`, `<a>`, `role="button"`,
   `onClick`, `<summary>`, `<select>`, label-for-checkbox) confirm it has the appropriate cue; add it where
   missing. Treat R19.2's audit matrix as untrusted and re-verify each row. Static/display elements stay
   untouched (Open Question 2 stands).

**State:** presentational only; no behavior, no DB. Preserve focus rings, `data-testid`s, `aria`.

**Open question — project/timeline status badges:** the deliverable status pill is already click-to-change;
the `ProjectStatusBadge`/`TimelineStatusBadge` spans are currently display-only. If project status should
*also* be changeable by clicking its badge, that's a **behavior change (new feature), not an affordance** —
wiring a status dropdown + a change action — and belongs outside R19.3's presentational scope. **Recommend:**
keep them display-only here (so R19.3 stays affordance-only) and, if wanted, track "click project status to
change" as a separate feature. Confirm before adding any pointer to those spans (adding a pointer without
wiring the dropdown would be a false affordance).
