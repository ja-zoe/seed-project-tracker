## Carry-over from Rev 3

### R3.4 known bugs (must fix before new R3.4 features)

**Bug 1 — Missing `zod` dependency (build-blocking)**

The `/api/chat` route imports `z` from `zod` directly, but `zod` is not in `package.json`. The AI SDK v7 bundles zod internally but does not re-export it as a peer at the consuming app level.

```
Module not found: Can't resolve 'zod'
./src/app/api/chat/route.ts:3:1
```

**Fix:** `pnpm add zod`. No code changes needed.

**Bug 2 — No API key validation on connect**

When a user pastes an API key in `ChatProviderSetup` and clicks "Connect", the key is accepted immediately without being tested against the actual provider. If the key is invalid, wrong, or for the wrong model, the first chat message fails silently or with a confusing error from the AI provider — and the UI may enter a broken state.

**Required behavior:** After the user clicks "Connect", make a lightweight test call to the provider before accepting the credential:
- For Anthropic/OpenAI/DeepSeek/Copilot: send a minimal `streamText` or `generateText` request (e.g., 1-token completion with the smallest available model) via the `/api/chat` route or a dedicated `/api/ai/verify` endpoint.
- For Google Gemini: same approach with `gemini-2.0-flash`.
- Show a spinner on the "Connect" button during verification.
- On failure: surface the provider's error message (e.g., "Invalid API key", "Quota exceeded") in the setup panel and do not store the credential.
- On success: proceed as today (store in sessionStorage, close setup panel, open chat).

The verification call should be server-side (route handler) so the API key never touches client-side fetch headers directly, and so error responses from providers are normalized before being shown to the user.

**Testing requirement:** Before any R3.4 follow-on work or new chatbot features are merged into `develop`, Claude must be given a real API key by the user and must verify end-to-end that:
1. The key can be entered and passes validation.
2. A chat message can be sent and a streamed response is received.
3. At least one tool call (e.g., `listProjects`) executes successfully and the result appears inline in the chat.

## Feature specs

### R4.0 — Fix R3.4 carry-over bugs

**R4.0a — Add `zod` to dependencies**

```bash
pnpm add zod
```

That is the entire fix. Verify the build passes after (`pnpm build` or `pnpm tsc --noEmit`).

**R4.0b — API key verification on connect**

Create a new route `src/app/api/ai/verify/route.ts`:

```ts
POST /api/ai/verify
Body: { provider: AiProvider, apiKey: string, model: string }
Response: { ok: true } | { ok: false, error: string }
```

- Requires session (`requireAuth`).
- Calls `generateText` (not `streamText`) from the `ai` package with `maxTokens: 1` and a simple prompt ("hi") using `resolveModel(provider, apiKey, model)`.
- Returns `{ ok: true }` on success or `{ ok: false, error: <provider error message> }` on failure.
- Catches and normalizes errors — never bubbles a raw stack trace.

Update `src/components/chat-provider-setup.tsx`:

- On "Connect" click, call `POST /api/ai/verify` before calling `onConnect`.
- Show a spinner on the button during verification.
- If `ok: false`, show the `error` string below the API key input (red, `text-[#A4503C]`) and do not proceed.
- If `ok: true`, call `onConnect` as before.

**Testing gate:** After implementing R4.0a + R4.0b, the implementer must:
1. Ask the user for a real API key.
2. Start the dev server.
3. Open the chat widget, connect the provider, confirm the verification call succeeds.
4. Send a chat message, confirm a streaming response arrives.
5. Ask the assistant to list projects, confirm the `listProjects` tool fires and results appear.
6. Only after all 5 steps pass may other R4 features begin.

## Open questions / decisions before implementing

1. **R4.0b verification endpoint:** Should the verification be a separate `/api/ai/verify` route, or inline in `/api/chat` with a `verify: true` flag in the body? Recommendation: separate route (cleaner separation, easier to call from setup panel without constructing a fake message array).

2. **R4.0b error normalization:** Provider error shapes differ. Recommendation: catch all errors, extract `.message` string if present, fall back to a generic "Could not verify key — check that it is correct and has sufficient quota."
