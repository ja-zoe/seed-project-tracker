# R17.2 — MCP OAuth 2.1 support (so ChatGPT web can connect) — via a managed provider

**Status:** planned — **review before implementing.** Approach decided: **managed OAuth provider** (dual-auth,
no DB change). Two prerequisites are the user's: pick the provider + create its account/keys, and deploy the
app to public HTTPS (ChatGPT can't reach `localhost`).
**Files (new/changed):** `src/app/api/mcp/route.ts` (validate OAuth tokens + `WWW-Authenticate`),
`src/app/.well-known/oauth-protected-resource/route.ts` (new), `src/app/api/mcp/authorize/route.ts` +
consent UI (new — the OAuth authorize entry that reuses the CAS session), `src/lib/mcp-oauth.ts` (new —
token verification + user mapping), env config. `changes/CONTEXT.md` "Standing decisions" updated.

## Spec

**Problem:** ChatGPT's custom MCP connectors accept only **OAuth** or **no-auth**; our `/api/mcp` uses a
static `Bearer <mcpToken>`, which only works with header-settable clients (Claude Code/Desktop, Cursor,
Codex). No-auth is unacceptable (it would expose project data to the public internet). So ChatGPT support
requires turning `/api/mcp` into an **OAuth-protected resource**.

**Decision (resolved): use a managed OAuth provider** — an "OAuth-provider-for-AI-agents / MCP" product
(**Stytch Connected Apps** or **WorkOS AuthKit** — confirm one at review; both have MCP support + free tiers
that cover a club-sized user base). The provider implements the hard, security-critical Authorization Server
(authorize/token endpoints, PKCE, dynamic client registration / CIMD, token issuance + rotation). We do **not**
hand-roll OAuth crypto. This is consistent with the app already delegating identity to CAS.

**The security crux — bind every token to a CAS-verified user (no impersonation).** The OAuth "login" step
must reuse the app's existing **Rutgers CAS** session, so the user identity in the issued token is a real,
CAS-verified netId — never a self-asserted email. Concretely:
1. ChatGPT (client) starts the OAuth flow → hits our authorize endpoint (provider-fronted).
2. Our authorize route **requires an authenticated app session** (`requireAuth()` → CAS login if absent).
   The logged-in SEED user is therefore CAS-verified.
3. A short **consent screen** ("Allow ChatGPT to access SEED Tracker as <you>?") → on approve, the provider
   issues an access token **bound to that SEED user id** (passed as the subject/claim).
4. ChatGPT calls `/api/mcp` with the token.

**Token validation (dual-auth) in `/api/mcp`:** `authenticate()` becomes: if the bearer matches a
`User.mcpToken` → existing path (local clients, unchanged). Else verify it as an OAuth access token via the
provider (JWKS signature check or token introspection) → extract the SEED user id/subject → load the user →
require `status === ACTIVE` → proceed with that user's existing RBAC (no tool/permission changes). On any
auth failure, return `401` with
`WWW-Authenticate: Bearer resource_metadata="https://<host>/.well-known/oauth-protected-resource"` so MCP
clients can discover the AS.

**Discovery metadata:** add `GET /.well-known/oauth-protected-resource` (RFC 9728) returning JSON pointing at
the provider's authorization server (`authorization_servers`, `resource`, supported scopes). The provider
serves its own `/.well-known/oauth-authorization-server`; we just reference it.

**Scopes:** start minimal — a single `mcp` scope (access the MCP tools as the authenticated user). RBAC is
still enforced per-user server-side, so scope granularity isn't load-bearing for safety in the MVP.

**Out of scope / explicitly deferred:** finer per-tool scopes; an in-app "connected apps" management list
(revocation is via the provider dashboard initially; a `McpConnection` table + UI is a clean follow-up).

**Effort & dependencies (logged per project norms):** adds one managed-auth SDK dependency — justified by
the explicit ChatGPT-support request and to avoid owning OAuth crypto. Requires a provider account + env
keys (user-provided) and a public HTTPS deploy. Realistically a multi-PR feature; sequence after R17.1.

## Tests

Merge gate for R17.2 (once the provider is chosen + app deployed):
- [ ] `pnpm build` clean.
- [ ] `GET /.well-known/oauth-protected-resource` returns valid metadata; unauthenticated `POST /api/mcp`
      returns `401` with the `WWW-Authenticate` resource-metadata header.
- [ ] **Regression (local):** static `Bearer <mcpToken>` still works — re-run the Set 11 MCP curl checks
      (initialize / tools/list / a read tool) unchanged.
- [ ] OAuth path (against a deploy): a provider-issued token is accepted, maps to the correct CAS-verified
      user, and that user's RBAC is enforced; a token for a non-ACTIVE/unmapped user → 401/403.
- [ ] **Impersonation guard:** a token cannot be obtained for a netId without passing CAS for that netId
      (the authorize route requires `requireAuth()`; the issued subject = the session user id).
- [ ] End-to-end: add the connector in ChatGPT Developer Mode → complete OAuth → list + call a read tool
      (manual; ChatGPT Business/Enterprise).
