# R17.2 — MCP OAuth 2.1 support (so ChatGPT web can connect)

**Status:** planned — **large, security-sensitive; approach decision required (see Open Q1 in `_set.md`)**
**Files (approach-dependent):** `src/app/api/mcp/route.ts` (token validation + `WWW-Authenticate`),
`src/app/.well-known/oauth-protected-resource/route.ts` (new), plus either provider config **or** a
self-hosted AS (`src/app/api/oauth/*` + new DB tables). `changes/CONTEXT.md` "Standing decisions" should
record the final auth design.

## Spec

**Problem / question:** The user asked whether the MCP server can be used from ChatGPT's web app. ChatGPT's
"custom MCP connectors" (Developer Mode, Business/Enterprise) only offer **OAuth** or **no authentication** —
there is no field to paste a static API token. Today `/api/mcp` authenticates with a **static Bearer token**
(`User.mcpToken`, looked up in `authenticate()`), which works only with clients that let you set a custom
`Authorization` header (Claude Code, Claude Desktop, Cursor, Codex CLI). So **ChatGPT web cannot connect
as-is.** It *can* be supported by making the endpoint an OAuth 2.1 protected resource.

**What the MCP authorization spec + ChatGPT require:**
- The MCP server acts as an **OAuth 2.0 Resource Server**. On unauthenticated calls it returns `401` with a
  `WWW-Authenticate: Bearer resource_metadata="…/.well-known/oauth-protected-resource"` header.
- A **Protected Resource Metadata** document (`/.well-known/oauth-protected-resource`, RFC 9728) that points
  clients to the **Authorization Server**.
- An **Authorization Server** implementing OAuth 2.1: `/.well-known/oauth-authorization-server` metadata,
  `authorize` + `token` endpoints, **PKCE**, and **dynamic client registration (RFC 7591)** or
  **Client-ID-Metadata-Documents (CIMD)** — ChatGPT supports CIMD with public-client (`none`) or
  `private_key_jwt` client auth, and refresh tokens (`offline_access`) so access survives expiry.
- The MCP route validates the presented **access token**, resolves it to a SEED user, and (unchanged)
  enforces that user's RBAC. HTTPS required.

**Approach (dual-auth, additive — no regression):**
1. **Keep the static-token path** for local clients: `authenticate()` still accepts `Bearer <mcpToken>`.
2. **Add an OAuth path:** if the bearer isn't a known `mcpToken`, validate it as an OAuth access token
   (introspect/verify per the chosen AS) → map to the user → proceed. 401s gain the `WWW-Authenticate`
   resource-metadata header.
3. **Add** `/.well-known/oauth-protected-resource` (and, if self-hosting, `/.well-known/oauth-authorization-server`).

**Authorization Server — the decision (Open Q1):**
- **(b) Managed provider (RECOMMENDED):** front auth with WorkOS AuthKit / Stytch Connected Apps / Auth0 /
  Clerk — they implement the AS, DCR/CIMD, PKCE, token issuance/rotation; we only host the resource-metadata
  doc and validate tokens (JWKS). **No app DB change.** Lowest security risk; the app already delegates
  identity (CAS), so adding a second managed AS is consistent. Mapping AS-identity → SEED user is by email
  (CAS netId email) — needs a deliberate trust/link step so an OAuth login can't impersonate a netId.
- **(a) Self-hosted minimal AS:** new additive tables `OAuthClient`, `OAuthAuthCode`,
  `OAuthAccessToken`/`OAuthRefreshToken`; implement authorize/token/registration with PKCE and short-lived
  JWT access tokens. Full control, **no new vendor**, but we own crypto + token rotation + security review.
  DDL is additive only (new tables) via `scripts/apply-schema.ts`.

**Why this is gated, not auto-built:** it adds an auth surface to a security-critical boundary, likely a new
dependency/vendor, and a user→identity mapping that must not allow impersonation. Per project norms
(no heavyweight deps without justification; risky work spec'd and left for the user), **confirm the approach
(provider vs self-host, and which provider) before implementation.** The static-token MCP keeps working for
local agents throughout.

**Interim answer for the user (no code):** today, use the MCP server with **Claude Code / Claude Desktop /
Cursor / Codex CLI** (paste the `/account` token as the `Authorization: Bearer` header). ChatGPT web support
arrives with this feature once the AS approach is chosen.
