**Status:** planned ‚Äî **review before implementing.** Approach decided: **managed provider ‚Üí
Stytch Connected Apps** (chosen 2026-06-29). Dual-auth, no DB change. Two prerequisites are the user's:
create the Stytch project + API keys, and deploy the app to public HTTPS (ChatGPT can't reach `localhost`).
**Files (new/changed):** `src/app/api/mcp/route.ts` (validate Stytch tokens + `WWW-Authenticate`),

`changes/CONTEXT.md` "Standing decisions" updated.

## Spec

**Problem:** ChatGPT's custom MCP connectors accept only **OAuth** or **no-auth**; our `/api/mcp` uses a
static `Bearer <mcpToken>`, which only works with header-settable clients (Claude Code/Desktop, Cursor,
Codex). No-auth is unacceptable (it would expose project data to the public internet). So ChatGPT support
requires turning `/api/mcp` into an **OAuth-protected resource**.

**Decision (resolved): Stytch Connected Apps.** Stytch's "Connected Apps" is purpose-built to turn an app
that already has its **own** login (our Rutgers CAS) into an OAuth provider for AI agents / MCP: we keep CAS
as the user login, and Stytch supplies the Authorization Server mechanics (authorize/token endpoints, PKCE,
dynamic client registration / CIMD, token issuance + rotation) and the consent + token validation
(JWKS/introspection). We do **not** hand-roll OAuth crypto. Free tier covers a club-sized user base.
*Alternatives considered:* WorkOS (equivalent "keep your login" model ‚Äî fine fallback); Supabase OAuth
Server (in-stack but ties token identity to **Supabase Auth** users, and we're CAS/Auth.js ‚Äî rejected to
avoid bridging identity stores). The `mcp-oauth.ts` seam keeps the provider swappable if that changes.

**The security crux ‚Äî bind every token to a CAS-verified user (no impersonation).** The OAuth "login" step
must reuse the app's existing **Rutgers CAS** session, so the user identity in the issued token is a real,
CAS-verified netId ‚Äî never a self-asserted email. Concretely:
1. ChatGPT (client) starts the OAuth flow ‚Üí hits our authorize endpoint (provider-fronted).
2. Our authorize route **requires an authenticated app session** (`requireAuth()` ‚Üí CAS login if absent).
   The logged-in SEED user is therefore CAS-verified.
3. A short **consent screen** ("Allow ChatGPT to access SEED Tracker as <you>?") ‚Üí on approve, the provider
   issues an access token **bound to that SEED user id** (passed as the subject/claim).
4. ChatGPT calls `/api/mcp` with the token.

**Token validation (dual-auth) in `/api/mcp`:** `authenticate()` becomes: if the bearer matches a
`User.mcpToken` ‚Üí existing path (local clients, unchanged). Else verify it as an OAuth access token via the
provider (JWKS signature check or token introspection) ‚Üí extract the SEED user id/subject ‚Üí load the user ‚Üí
require `status === ACTIVE` ‚Üí proceed with that user's existing RBAC (no tool/permission changes). On any
auth failure, return `401` with
`WWW-Authenticate: Bearer resource_metadata="https://<host>/.well-known/oauth-protected-resource"` so MCP
clients can discover the AS.

**Discovery metadata:** add `GET /.well-known/oauth-protected-resource` (RFC 9728) returning JSON pointing at
the provider's authorization server (`authorization_servers`, `resource`, supported scopes). The provider
serves its own `/.well-known/oauth-authorization-server`; we just reference it.

**Scopes:** start minimal ‚Äî a single `mcp` scope (access the MCP tools as the authenticated user). RBAC is
still enforced per-user server-side, so scope granularity isn't load-bearing for safety in the MVP.

**Out of scope / explicitly deferred:** finer per-tool scopes; an in-app "connected apps" management list
(revocation is via the provider dashboard initially; a `McpConnection` table + UI is a clean follow-up).

**Effort & dependencies (logged per project norms):** adds one managed-auth SDK dependency ‚Äî justified by
the explicit ChatGPT-support request and to avoid owning OAuth crypto. Requires a provider account + env
keys (user-provided) and a public HTTPS deploy. Realistically a multi-PR feature; sequence after R17.1.

## Build phases

**Phase A ‚Äî resource-server foundation (DONE, verified locally 2026-06-29):** added the `stytch` SDK,
`src/lib/mcp-oauth.ts` (Stytch client + `verifyOAuthAccessToken` via `idp.introspectTokenLocal`, mapping the
token's CAS email ‚Üí ACTIVE SEED user, behind a swappable seam), `/.well-known/oauth-protected-resource`
metadata route, and dual-auth in `/api/mcp` (static `mcpToken` first by JWT-shape, else Stytch introspection)
with a `WWW-Authenticate` resource-metadata 401.

**Trusted Auth Token Profile ‚Äî canonical config (the code in Phase B MUST match these):**
| Stytch dashboard field | Value | Code side |
|---|---|---|
| Name | `SEED Tracker (CAS)` | ‚Äî |
| Issuer | the deployed app base URL (= `AUTH_URL`) | minter sets JWT `iss` to `AUTH_URL` |
| Audience | the **Stytch Project ID** (`STYTCH_PROJECT_ID`) | minter sets JWT `aud` to `STYTCH_PROJECT_ID` |
| Allow JIT Provisioning | **ON** (CAS users don't pre-exist in Stytch) | ‚Äî |
| JWKS | `<AUTH_URL>/.well-known/jwks.json` | new route serves the public JWK |
| Required: email | `email` | minter puts CAS email in the `email` claim |
| Required: token_id | `sub` | minter puts the SEED user id in `sub` |
| Optional: external_user_id / role_ids | blank | RBAC stays server-side in the app |

The trusted token (JWT #1) is **asymmetrically signed** (RS256) ‚Äî private key in env (`MCP_OAUTH_PRIVATE_KEY`),
public key published at `/.well-known/jwks.json`. (This is distinct from the app's HMAC `handoff-token`.)

**Phase B ‚Äî authorization flow (REMAINING; needs the Stytch dashboard + a public HTTPS deploy):**
- `/oauth/authorize` route: `requireAuth()` (CAS) ‚Üí mint a JWT containing the user's `email` ‚Üí render Stytch's
  `IdentityProvider` component with `trustedAuthToken={jwt}` (needs the `@stytch/nextjs` frontend SDK + a
  `StytchProvider` scoped to this route, using `STYTCH_PUBLIC_TOKEN`).
- **Stytch dashboard (user):** create a **Trusted Auth Token Profile** that validates our minted JWT and
  passes `email` through as a custom claim; set the **Authorization URL** to the deployed `/oauth/authorize`;
  copy the AS issuer into `STYTCH_OAUTH_AUTHORIZATION_SERVER`.
- **Deploy (user):** public HTTPS (e.g. Vercel) ‚Äî required for ChatGPT + Stytch to reach the endpoints.
