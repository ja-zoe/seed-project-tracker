# R13.2 — Fix missing `SUBLEAD` value in the DB `ProjectMemberRole` enum

**Status:** done
**Files:** none in source — **DB-only** change applied via `scripts/apply-schema.ts`

## Spec

**Problem:** Adding someone to a project as a **sub-lead** throws at runtime:

```
PrismaClientKnownRequestError: Invalid `prisma.projectAssignment.upsert()` invocation:
invalid input value for enum "ProjectMemberRole": "SUBLEAD"
    at assignMember (src/lib/actions/projects.ts:103)
```

**Root cause (verified):** the Prisma schema declares `enum ProjectMemberRole { LEAD SUBLEAD MEMBER }`
and the generated client knows all three, but the **live Postgres enum only has `LEAD` and `MEMBER`** —
the `SUBLEAD` value was never applied to the database (a schema-vs-DB drift, the same class of issue noted
in `CONTEXT.md` / the prisma-v7 memory). The members form offers "Sub-lead", so picking it sends
`"SUBLEAD"`, which the DB rejects.

Confirmed via `pg_enum`: `ProjectMemberRole` → `['LEAD', 'MEMBER']`.

**Approach:** Add the missing value to the database enum. **No `schema.prisma` change and no
`prisma generate`** — the schema and client already have `SUBLEAD`; only the DB is behind. Apply
statement-by-statement (an `ALTER TYPE … ADD VALUE` cannot run inside a transaction — see Set 10/11):

```sql
ALTER TYPE "ProjectMemberRole" ADD VALUE IF NOT EXISTS 'SUBLEAD' BEFORE 'MEMBER';
```

Write that single statement to `/tmp/schema.sql` and run `tsx scripts/apply-schema.ts`. `BEFORE 'MEMBER'`
keeps the DB sort order matching the schema (LEAD, SUBLEAD, MEMBER); functionally any position works.

**Why it unblocks other work:** R13.5 (edit member role) and the existing "Add member" form both depend on
`SUBLEAD` being a valid DB value. This must land before R13.5 is verified. After applying, restart the dev
server is **not** required (no client/WASM regeneration — the client already had the value).
