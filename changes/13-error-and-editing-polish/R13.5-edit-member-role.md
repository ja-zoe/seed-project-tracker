# R13.5 — Edit a project member's role

**Status:** tests passing
**Files:** `src/app/(app)/projects/[id]/members/page.tsx`, `src/components/member-role-control.tsx` (new
client island), `src/lib/actions/projects.ts` (reuse `assignMember`, or add a thin `updateMemberRole`).
Depends on **R13.2** (the DB must have `SUBLEAD`).

## Spec

**Problem:** On `/projects/<id>/members`, each current member shows their role as static text
(`{a.role.toLowerCase()}`); there's no way to **change** an existing member's project role (LEAD / SUBLEAD /
MEMBER) — you can only remove and re-add them. Privileged users should be able to edit the role in place.

**Approach:**

*Server action.* `assignMember(projectId, userId, role)` already `upsert`s with `update: { role }`, so it
can change an existing member's role as-is. Either reuse it directly, or add a small, clearly-named
`updateMemberRole(projectId, userId, role)` wrapper (same `requirePermission(MANAGE_PROJECTS)` + update +
`revalidatePath`). Recommendation: add `updateMemberRole` for clarity and so it can
`revalidatePath(\`/projects/${projectId}/members\`)` (the current `assignMember` only revalidates the
project page). Gate by **`MANAGE_PROJECTS`** (Open Q1) — the members page already requires it.

*UI.* Replace the static role text in each current-member row with a Client island
`src/components/member-role-control.tsx`: shows the role; clicking it reveals an inline `<select>`
(Lead / Sub-lead / Member) plus an `InlineConfirm` ✓/✗ (Open Q4 — same microinteraction as R13.4 for
consistency). ✓ calls the server action with the new role (via `useTransition`); ✗/Escape cancels. Props:
`{ projectId, userId, role }`. The members page (server component) renders `<MemberRoleControl … />` in
place of `{a.role.toLowerCase()}`.

*Edge cases:* no-op if the role is unchanged (skip the write). Don't restrict removing the last LEAD in
this set unless the user asks (out of scope). The "Add member" form already offers Sub-lead — once R13.2
lands, both add and edit can set SUBLEAD.

**Why R13.2 first:** selecting "Sub-lead" writes `"SUBLEAD"`; without R13.2 the DB rejects it with the same
enum error this set fixes.
