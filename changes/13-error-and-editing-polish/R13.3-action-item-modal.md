# R13.3 — Action-item create/edit in a modal

**Status:** planned
**Files:** `src/components/action-item-modal.tsx` (new), `src/components/action-items-section.tsx` (new,
client island extracted from the project page), `src/app/(app)/projects/[id]/page.tsx`,
`src/lib/actions/action-items.ts` (drop the redirect in `updateActionItem`),
**delete** `src/app/(app)/projects/[id]/action-items/[aid]/edit/page.tsx`
(optional: `src/app/(app)/action-items/page.tsx` to reuse the modal — see Open Q2)

## Spec

**Problem:** Action-item **creation** is a cramped inline form on the project page
(`page.tsx` ~308–348: description input + owner `<select>` + date + "+" button), and **editing** is a
whole separate route (`action-items/[aid]/edit/page.tsx`) that `updateActionItem` `redirect`s back from.
The user wants full create **and** edit in a **modal**, consistent with deliverables/subtasks
(`deliverable-modal.tsx`, `subtask-modal.tsx`, set 8/9), not a separate page.

**Approach:**

*Modal component* — `src/components/action-item-modal.tsx`, a Client Component mirroring
`subtask-modal.tsx`: built on `@/components/ui/dialog` (`Dialog`/`DialogTrigger`/`DialogContent`/
`DialogHeader`/`DialogTitle`), `useState` for open, `useTransition` for the submit. Props:
`{ projectId: string; assignees: { id: string; name: string }[]; item?: ActionItemDTO; trigger: ReactElement }`.
Fields = the action-item fields: **description** (text/textarea, required), **owner** (`<select>` over
`assignees`, "No owner" option), **deadline** (`type="date"`, optional). On submit:
- create mode (`!item`): build `FormData` → `await createActionItem(projectId, fd)`;
- edit mode: `await updateActionItem(item.id, fd)`.
Close on success. Reuse field styling + `SubmitButton` conventions from the existing edit page/subtask modal.

*Server action change* — in `action-items.ts`, **remove the `redirect(\`/projects/…\`)`** at the end of
`updateActionItem` (and the now-unused `redirect` import if nothing else needs it); keep the
`revalidatePath` calls. The redirect only existed to leave the standalone edit page; from a modal we stay
in place and let revalidation refresh the list. `createActionItem` already just revalidates — unchanged.

*Project page* — extract the action-items block (the "Action Items" `<section>`, ~287–464) into a Client
island `src/components/action-items-section.tsx` that receives serializable props: `projectId`,
`items` (id, description, ownerId, owner name, deadline ISO, status, carriedOver), `assignees`
(project assignments → id/name), and the permission booleans (`canCreate`, `canClose`, `currentUserId`).
It renders: a **"New action item"** button that opens `ActionItemModal` in create mode (replacing the
inline create form); the open/done lists; per row an **Edit** button that opens `ActionItemModal` in edit
mode (replacing the `<Link href=…/edit>`); and the existing close/reopen controls (call the
`closeActionItem`/`reopenActionItem` server actions directly from the client island — they're `"use server"`).
The project page (server component) computes the props and renders `<ActionItemsSection … />`.
**This section becomes the shared host for R13.4's inline edits** — build it with that in mind.

*Delete the edit route* — remove `src/app/(app)/projects/[id]/action-items/[aid]/edit/page.tsx`; nothing
links to it after the project page switches to the modal (verify no other references).

*Global list (Open Q2)* — optionally give `/action-items/page.tsx` the same modal for editing; otherwise
leave it close/reopen-only this set.

**RBAC (unchanged):** create/edit gated by project lead/sublead membership or `ASSIGN_ACTION_ITEMS`
(as `createActionItem`/`updateActionItem` already enforce server-side); close by owner or `CLOSE_ACTION_ITEMS`.
The client only hides/shows controls; the server actions remain the source of truth.
