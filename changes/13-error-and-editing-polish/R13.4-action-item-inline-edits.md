# R13.4 — Action-item inline per-field edits with confirmation microinteractions

**Status:** tests passing
**Files:** `src/components/action-items-section.tsx` (the client island from R13.3), possibly a small
`src/components/action-item-row.tsx` split out of it. Reuses `InlineConfirm` from
`src/components/sortable-deliverables.tsx`. Depends on **R13.3**.

## Spec

**Problem:** Beyond the full-edit modal, the user wants **quick inline edits** on each action-item row,
with the **same confirmation microinteractions** used elsewhere — i.e. the deliverable/subtask pattern from
sets **7** (`changes/7-inline-edit-polish`) and **9** (`changes/9-deliverable-editing`): click a field to
edit it in place, then a sliding **✓ / ✗** `InlineConfirm` commits or cancels. This is faster than opening
the modal for a one-field tweak and keeps action items consistent with deliverables/subtasks.

**Approach:** Build on the R13.3 client island. For each **open** action item, make the three fields
inline-editable (done items stay read-only):

- **Description** — click the text → it becomes an `<input>`/`<textarea>` seeded with the current value;
  an `InlineConfirm` appears. ✓ commits, ✗/Escape/blur-cancel reverts.
- **Owner** — click the owner chip → inline `<select>` over the project assignees + "No owner";
  `InlineConfirm` to commit.
- **Deadline** — click the due chip → inline `type="date"`; `InlineConfirm` to commit (and a way to clear).

State lives in the row component: `editingField: "description" | "owner" | "deadline" | null` (+ a draft
value), exactly like `sortable-deliverables.tsx`'s inline-edit state. Commit calls **`updateActionItem`**
(made redirect-free in R13.3) with a `FormData` carrying **all three current values** with the edited field
overridden (the action updates description/ownerId/deadline together), wrapped in `useTransition`; on
success the field returns to display mode and `revalidatePath` refreshes. Reuse the shared
`InlineConfirm` (import from `sortable-deliverables.tsx`; props `show/onConfirm/onCancel/disabled`) and its
CSS slide transition — do **not** re-implement the microinteraction.

**Consistency rules (match the precedent):**
- Only one field per row in edit mode at a time; opening another cancels the first.
- ✗ / Escape cancels without writing; clicking ✓ (or Enter on a single-line field) commits.
- Disable the ✓/✗ while the transition is pending (`isPending`), same as deliverables.
- Permission gating identical to the modal edit (lead/sublead or `ASSIGN_ACTION_ITEMS`); rows without
  permission render fields read-only (no click-to-edit).

**Relationship to R13.3:** the modal stays the path for *create* and *multi-field* edits; inline editing is
the quick single-field path. Same row component hosts both (an "Edit" button → modal; field click → inline).
