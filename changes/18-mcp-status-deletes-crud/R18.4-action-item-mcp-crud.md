# R18.4 — Full action-item CRUD over MCP (`delete_action_item`)

**Status:** tests passing (e2e green) — awaiting merge.
**Files:** `src/app/api/mcp/route.ts` (TOOLS array + new handler `case`),
`src/app/(app)/account/mcp-token-section.tsx` (tool list display). Depends on R18.3's
`deleteActionItem` permission logic for parity (but the MCP route checks perms inline, see below).

## Spec

**Problem:** The MCP server exposes `create_action_item`, `update_action_item` (close/reopen/edit), and
`list_action_items`, but **not delete** — so action-item CRUD over MCP is incomplete. The request: expose
full CRUD, gated by the appropriate permissions.

**Approach:**

1. **Audit the CRUD surface.** Create ✓ (`create_action_item`), Read ✓ (`list_action_items`,
   `get_project_detail` includes open items), Update ✓ (`update_action_item` covers close/reopen/edit).
   **Delete is the only gap** → add `delete_action_item`.

2. **New tool.** Add `delete_action_item` to the `TOOLS` array with an input schema of
   `{ action_item_id: string }` and a description noting the permission ("Delete an action item. Requires
   CLOSE_ACTION_ITEMS, ASSIGN_ACTION_ITEMS, or project LEAD/SUBLEAD."). Add a `case "delete_action_item"`
   handler mirroring `update_action_item`'s gating exactly (lines around the existing
   `case "update_action_item"`): load the item → `getProjectMembership(user.id, actionItem.projectId)` →
   allow if `permissions.includes(CLOSE_ACTION_ITEMS) || permissions.includes(ASSIGN_ACTION_ITEMS) ||
   membership.role ∈ {LEAD, SUBLEAD}` → `prisma.actionItem.delete` → return a small JSON confirmation
   (`{ deleted: true, id }`) in the standard `{ content: [{ type: "text", text: … }] }` envelope. Return the
   route's standard error shape (not a throw) when not found / not permitted, matching `delete_deliverable`.

3. **Tool-count + docs.** The MCP server currently advertises 24 tools (see R17.2 regression note); adding
   one makes 25 — update any hard-coded count and bump `serverInfo.version` if that's the convention.
   In `mcp-token-section.tsx`, add `delete_action_item` to the "Action items" group of the displayed tool
   list ("delete an action item (LEAD/SUBLEAD or PM)").

**State:** none beyond the DB mutation. **RBAC:** enforced inline in the route handler (the MCP route does
not call the server actions; it re-checks permissions itself — keep parity with R18.3's gate so UI and MCP
agree). **Sequencing:** R18.1 also edits `api/mcp/route.ts` and `mcp-token-section.tsx` (different regions) —
do R18.1 first to avoid a merge conflict.

**Edge cases:** unknown `action_item_id` → standard "not found" error result (not a 500). A token whose user
lacks the capability → permission-denied error result. Deletion via MCP must revalidate nothing (no Next
cache in the API route), but the next UI load reflects it.
