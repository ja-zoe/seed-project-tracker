# R18.2 — Delete active users (permission + UI with confirm microinteraction)

**Status:** planned — **review before implementing** (see Open Question 2 in `_set.md`).
**Files:** `prisma/schema.prisma` (Permission enum + two FK `onDelete`), DDL via `scripts/apply-schema.ts`,
`src/lib/actions/users.ts` (new `deleteUser`), `src/app/(app)/pm/users/page.tsx`, a new client component
`src/components/user-row-controls.tsx` (or `delete-user-button.tsx`), `prisma/seed*` / `db:seed` (grant the
new permission to "Project Manager"), and the `ALL_PERMISSIONS` list in `pm/users/page.tsx`.

## Spec

**Problem:** PMs can approve, suspend, and reactivate users (`actions/users.ts`), but cannot remove a user
entirely. There's no permission for it and no UI. The request: a dedicated permission plus a PM control to
**delete active users**, with a confirmation microinteraction before the delete completes.

**Approach (hard-delete + block-on-history — confirm Open Question 2):**

1. **Permission.** Add `DELETE_USERS` to the `Permission` enum.
   DDL: `ALTER TYPE "Permission" ADD VALUE IF NOT EXISTS 'DELETE_USERS';` (own statement, before any use).
   Add to `ALL_PERMISSIONS` in `pm/users/page.tsx` (`{ value: Permission.DELETE_USERS, label: "Delete users" }`)
   and grant it to the built-in **Project Manager** role in `db:seed`. Existing PM rows in the dev DB need a
   one-off update to add the permission (note in `_set.md` DB section).

2. **FK safety (DDL).** Make the optional author FKs null out instead of RESTRICT on delete:
   - `Subtask.assignee` (`assigneeId`) → `ON DELETE SET NULL`
   - `ActionItem.owner` (`ownerId`) → `ON DELETE SET NULL`
   Reflect `onDelete: SetNull` in `schema.prisma` and apply the constraint swap via raw DDL (drop + re-add
   the FK). `Account/Session/ProjectAssignment/Notification` already cascade.

3. **Server action.** `deleteUser(userId: string)` in `actions/users.ts`:
   - `await requirePermission(Permission.DELETE_USERS)`.
   - Guard: refuse to delete **yourself** (compare to `requireAuth()` user id).
   - Guard (history): if the user has any `StatusUpdate` or `MeetingRecord` rows (required-author records
     that can't be null'd), throw a friendly error ("This user has authored history — suspend them instead")
     so the UI can surface it. (Optionally also refuse if they're the **sole LEAD** of a project.)
   - Otherwise `prisma.user.delete({ where: { id } })` — cascades handle assignments/notifications/sessions;
     SetNull handles owned action items / assigned subtasks.
   - `revalidatePath("/pm/users")`.

4. **UI control.** The PM users page is a server component using inline server-action `<form>`s. Add a
   **client** component `UserRowControls` (mirroring `MeetingRecordControls`) that renders a "Delete" button
   which, on click, swaps to a "Delete? ✓ ✗" using the shared `InlineConfirm` from
   `@/components/sortable-deliverables`, calling `deleteUser(u.id)` in a `useTransition`. Place it in the
   active-users row next to Suspend. On the history-block error, show the returned message inline
   (catch + small `text-[#A4503C]` note) rather than crashing.

**State:** confirm/pending state is local to the `UserRowControls` client component; the mutation is the
server action; the list re-renders via `revalidatePath`. **RBAC:** server action is the source of truth
(`requirePermission(DELETE_USERS)`); the button is additionally only rendered when the viewing PM has the
permission (page already computes `getUserPermissions`).

**Non-obvious constraints:** never allow self-deletion; `ALTER TYPE … ADD VALUE` must precede any statement
that references it; existing Project Manager role rows must be back-filled with the new permission or the
PM loses access to their own new control until re-seeded.

## Tests

- [ ] `pnpm build` clean; `prisma generate` ok; DDL applied (enum value present, both FKs show
      `ON DELETE SET NULL` in `\d`).
- [ ] PM role has `DELETE_USERS` after `db:seed` (and the back-fill update ran for existing rows).
- [ ] A user with **no authored history** can be deleted: click Delete → inline "Delete?" confirm → confirm
      → row disappears; their `ProjectAssignment`/`Notification` rows are gone and any owned action
      items/assigned subtasks now have null owner/assignee (not deleted).
- [ ] A user **with** a StatusUpdate or MeetingRecord → delete is refused with the "suspend instead"
      message; the user remains.
- [ ] Self-delete is refused.
- [ ] A non-PM (or PM role lacking `DELETE_USERS`) gets no Delete control and the action throws if called.
- [ ] Cancel (✗) on the confirm leaves the user untouched.
