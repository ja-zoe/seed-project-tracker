# R18.2 — Delete active users (permission + UI with confirm microinteraction)

**Status:** done — merged to main (local, not pushed) 2026-06-30.
**Files:** `prisma/schema.prisma` (`Permission` enum + `UserStatus` enum), DDL via `scripts/apply-schema.ts`,
`src/lib/actions/users.ts` (new `deleteUser`), `src/app/(app)/pm/users/page.tsx`, a new client component
`src/components/user-row-controls.tsx`, `src/auth.ts` / `src/lib/permissions.ts` (treat `DELETED` as
non-active), `db:seed` (grant the new permission to "Project Manager"), and `ALL_PERMISSIONS` in
`pm/users/page.tsx`.

## Spec

**Problem:** PMs can approve, suspend, and reactivate users (`actions/users.ts`), but cannot remove a user.
There's no permission for it and no UI. The request: a dedicated permission plus a PM control to delete
active users, with a confirmation microinteraction before the delete completes.

**Approach (decided: anonymizing soft-delete).** We do **not** hard-delete the row — that would orphan or
destroy authored history (StatusUpdates, MeetingRecords, owned ActionItems, assigned Subtasks). Instead we
mark the user `DELETED` and scrub their PII, keeping all FKs/history intact. This avoids any FK/DDL surgery
on related tables.

1. **Permissions / statuses (DDL).**
   - `ALTER TYPE "Permission" ADD VALUE IF NOT EXISTS 'DELETE_USERS';`
   - `ALTER TYPE "UserStatus" ADD VALUE IF NOT EXISTS 'DELETED';`
   Both in their own statements, before any use (see `_set.md` note). Reflect both in `schema.prisma`.
   Add `{ value: Permission.DELETE_USERS, label: "Delete users" }` to `ALL_PERMISSIONS` in `pm/users/page.tsx`,
   grant `DELETE_USERS` to the built-in **Project Manager** role in `db:seed`, and back-fill existing PM
   role rows in the dev DB (one-off update — note in `_set.md`). **No FK changes needed** (the row stays).

2. **Server action.** `deleteUser(userId: string)` in `actions/users.ts`:
   - `await requirePermission(Permission.DELETE_USERS)`.
   - **Hard guard: never self-delete.** Compare to `requireAuth()` user id and refuse (the request
     explicitly calls out that a PM must never be able to delete themselves). Throw a clear error.
   - Anonymize + soft-delete in one update: `status = DELETED`; `name/firstName/lastName/nickname` →
     null (or display "Deleted user"); `email` → a unique placeholder so it can't collide and the original
     netId can re-register later (e.g. `deleted+<cuid>@seed.invalid` — keeps the `@unique` constraint
     happy); `mcpToken` → null; `image` → null.
   - Revoke access: delete the user's `Session` and `Account` rows (force logout) and their
     `ProjectAssignment` rows (drop project access). **Keep** StatusUpdate/MeetingRecord/ActionItem/Subtask
     links — they now point at the anonymized user, preserving history.
   - `revalidatePath("/pm/users")`.

3. **Auth treats `DELETED` as inactive.** Audit where `status === "ACTIVE"` / `!== "ACTIVE"` is checked
   (`src/auth.ts`, `src/lib/permissions.ts`, `/api/mcp`): a `DELETED` user is not ACTIVE, so they're already
   locked out — confirm none of these accidentally treat non-ACTIVE as "re-activatable" for DELETED. A
   returning netId signs in fresh as a new PENDING user (their old email was changed).

4. **UI control.** The PM users page is a server component using inline server-action `<form>`s. Add a
   **client** component `UserRowControls` (mirroring `MeetingRecordControls`) with a "Delete" button that
   swaps to "Delete? ✓ ✗" via the shared `InlineConfirm` from `@/components/sortable-deliverables`, calling
   `deleteUser(u.id)` in a `useTransition`. Place it next to Suspend on the active-users row. Hide it on the
   PM's own row (no self-delete). Filter `DELETED` users out of the main list (or show them in a muted,
   collapsed "Deleted" section — pick the list filter for simplicity). Surface any thrown error inline
   (`text-[#A4503C]`).

**State:** confirm/pending is local to `UserRowControls`; the mutation is the server action; the list
re-renders via `revalidatePath`. **RBAC:** server action is the source of truth
(`requirePermission(DELETE_USERS)`); the button renders only when the viewing PM has the permission (page
already computes `getUserPermissions`) and never on their own row.

**Non-obvious constraints:** the email rewrite is what lets a deleted netId sign up again and avoids a unique
collision; `ALTER TYPE … ADD VALUE` must precede any use; back-fill existing PM rows or the PM can't see
their own new control until re-seeded; soft-delete preserves history by design (that was the whole point of
choosing it over hard-delete).

      present — verified in `pg_enum`).
- [x] PM role has `DELETE_USERS` (verified in DB after the back-fill `array_append`; `seed.ts` updated for
      future seeds).

      `deleted+<id>@seed.invalid`, null mcpToken, and 0 Session/ProjectAssignment rows.
- [ ] History preserved (StatusUpdate/MeetingRecord/ActionItem still reference the anonymized user) — by

      in e2e (victim had no authored history).
- [x] **Self-delete is always refused** — no Delete control on the PM's own active row (e2e); the action
      also hard-guards `me.id === userId` and throws (code).
- [ ] A `DELETED` user cannot sign in / re-registers fresh as PENDING — implemented in `requireAuth` +
      `auth.ts`; not separately e2e-tested.
- [ ] A non-PM (or PM lacking `DELETE_USERS`) gets no control and the action throws — control gated on
      `canDeleteUsers`; action calls `requirePermission(DELETE_USERS)`. Not e2e-tested (no non-PM fixture).

      not separately asserted in the user spec.
