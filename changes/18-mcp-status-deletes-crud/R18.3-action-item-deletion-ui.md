# R18.3 — Action-item deletion in the UI (inline + modal)

**Status:** planned.
**Files:** `src/lib/actions/action-items.ts` (new `deleteActionItem`),
`src/components/action-items-section.tsx` (inline row control, open + closed lists),
`src/components/action-item-modal.tsx` (modal delete control).

## Spec

**Problem:** Action items can be created, edited (inline + modal), closed, and reopened, but **not
deleted**. A mistaken or obsolete item lingers; the only "removal" is closing it. The request: delete an
action item inline from its row, and also from the edit modal, both using the same confirmation
microinteraction already used elsewhere (the shared `InlineConfirm`).

**Approach:**

1. **Server action.** `deleteActionItem(actionItemId: string)` in `actions/action-items.ts`, mirroring the
   permission model of `updateActionItem`:
   - load the item → resolve `projectId`;
   - `requireAuth()` + allow if `getUserPermissions` includes `ASSIGN_ACTION_ITEMS` **or** the user's
     `getProjectMembership` role is `LEAD`/`SUBLEAD` (same gate as edit). (Reuse the exact check
     `updateActionItem` uses so the rules don't drift.)
   - `prisma.actionItem.delete(...)`;
   - `revalidatePath` for the project's action-items view **and** the global `/action-items` page (both list
     it). Match the paths `updateActionItem`/`closeActionItem` already revalidate.

2. **Inline (row) control.** In `action-items-section.tsx`'s `ActionItemRow` (open items) add a Delete
   control alongside the existing edit/close affordances, using the shared `InlineConfirm` pattern exactly
   like `MeetingRecordControls`: a "Delete" button that swaps to "Delete? ✓ ✗", calling `deleteActionItem`
   in the row's existing `useTransition`. Add the same control to the **closed/reopen** list rows so closed
   items can also be deleted. Use the clay tone for the trigger (`hover:text-[#A4503C]`).

3. **Modal control.** In `action-item-modal.tsx`, when `mode === "edit"`, add a Delete affordance to the
   footer (left side, separated from Cancel/Save) using the same `InlineConfirm` microinteraction. On
   confirm: call `deleteActionItem(item.id)`, then close the modal. Guard so it only renders in edit mode
   (no item to delete in create mode).

**State:** confirm/pending is local component state (`useState` + `useTransition`) in each location, same as
the existing inline editors. Data flow: click → server action → `prisma.delete` → `revalidatePath` → list
re-renders. **RBAC** enforced in the action; the controls render only when the row's existing `canEdit`/
`canClose` capability is true (already computed and passed into the row/modal).

**Edge cases:** deleting a carried-over item removes only that instance. Deleting the last open item should
leave the section's empty state intact (revalidate handles it). Don't show the delete control to viewers
without edit/close capability.

## Tests

- [ ] `pnpm build` clean.
- [ ] Inline: as a LEAD/PM, an open action item shows Delete → confirm → it disappears; cancel leaves it.
- [ ] Inline on a **closed** item: Delete works there too.
- [ ] Modal: open edit modal → Delete (with confirm) removes the item and closes the modal.
- [ ] A member without edit/close capability sees no Delete control, and `deleteActionItem` throws if
      invoked directly.
- [ ] Both the project action-items view and the global `/action-items` page reflect the deletion (revalidate).
