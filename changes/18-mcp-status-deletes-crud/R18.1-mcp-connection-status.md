# R18.1 — MCP connection status on the account page

**Status:** planned — **approach decided (Open Question 1 → `McpConnection` table, user 2026-06-30).**
**Files:** `prisma/schema.prisma` (new `McpConnection` model + `McpConnectionType` enum), DDL via
`scripts/apply-schema.ts`, `src/app/api/mcp/route.ts` (`authenticate()` upserts a connection),
`src/lib/mcp-oauth.ts` (return the OAuth client id/label for the upsert),
`src/lib/actions/account.ts` (`generateMcpToken`/`revokeMcpToken` maintain the access-token connection),
`src/app/(app)/account/page.tsx`, `src/app/(app)/account/mcp-token-section.tsx`,
`src/lib/utils.ts` (relative-time helper if not present).

## Spec

**Problem:** The account page's MCP section shows only a static `Token active` / `No token` pill, derived
from whether `User.mcpToken` exists. That tells the user nothing about whether an AI client is *actually*
connected, and it cannot represent an **OAuth (ChatGPT)** connection at all — OAuth access tokens are
Stytch-managed and leave no local record (R17.2 deferred a `McpConnection` table). So a user who connected
ChatGPT sees "No token" and assumes nothing works; a user with a stale token can't tell it's unused.

**Approach (decided: a dedicated `McpConnection` table):** record each MCP connection as a row, with its
type, optional OAuth client, and a `lastSeenAt` that `/api/mcp` updates on use. This also lays the
groundwork for a future "connected apps" management/revoke UI (the R17.2 follow-up).

1. **Schema.** New enum + model:
   ```prisma
   enum McpConnectionType { ACCESS_TOKEN OAUTH }

   model McpConnection {
     id         String            @id @default(cuid())
     userId     String
     user       User              @relation(fields: [userId], references: [id], onDelete: Cascade)
     type       McpConnectionType
     clientId   String?           // OAuth client id (from the token); null/sentinel for personal token
     label      String?           // friendly name, e.g. "ChatGPT" or "Personal access token"
     createdAt  DateTime          @default(now())
     lastSeenAt DateTime          @default(now())

     @@unique([userId, type, clientId])   // one row per (user, type, client)
   }
   ```
   Add `mcpConnections McpConnection[]` to `User`. DDL: create the enum, the table, the FK (ON DELETE
   CASCADE), and the unique index, via `tsx scripts/apply-schema.ts`; then `prisma generate` + **restart
   dev** (Turbopack WASM cache). **Note:** Postgres treats NULLs as distinct in a unique index, so for
   `ACCESS_TOKEN` use a fixed sentinel `clientId` (e.g. `"personal"`) rather than null, so the upsert keys
   to a single row. (OAuth rows always carry a real client id.)

2. **Upsert on use.** In `src/app/api/mcp/route.ts` `authenticate()`, after a successful resolution, upsert
   the connection and bump `lastSeenAt = now()` (best-effort, wrapped in `try/catch` — a write failure must
   never 500 a tool call; throttle to ~once/5min by skipping the write if `lastSeenAt` is recent):
   - static-token branch → `{ type: ACCESS_TOKEN, clientId: "personal", label: "Personal access token" }`.
   - OAuth branch → `{ type: OAUTH, clientId: <token client id>, label: <client name or "OAuth client"> }`.
     Extend `verifyOAuthAccessToken` to also return the token's client id (the JWT carries it — e.g. the
     `client_id`/`azp`/`aud` claim seen in R17.2 as `connected-app-…`) and a label if available, so
     `authenticate()` can key the upsert.

3. **Maintain the access-token connection in `account.ts`.** `generateMcpToken()` upserts the
   `ACCESS_TOKEN` connection (createdAt now, label "Personal access token"); `revokeMcpToken()` deletes the
   `ACCESS_TOKEN` connection row (so revoking clears the status immediately, not just on next use).

4. **Read on the account page.** `src/app/(app)/account/page.tsx` loads the user's `McpConnection` rows
   (ordered by `lastSeenAt desc`) and passes them to `MpcTokenSection` as a typed list
   (`{ type, label, lastSeenAt }[]`), alongside the existing `hasToken`.

5. **Render status.** In `mcp-token-section.tsx`, replace the single status pill with a "Connections" list:
   one line per `McpConnection` — icon + `label` + "last used {relative}", with a green dot when
   `lastSeenAt` ≤ 7d (else muted). If there are no connections, show "No active connections". Keep the
   existing generate/regenerate/revoke controls below. Copy says "last used" (not "currently connected") —
   we detect activity, not live/remote-revoked sessions.

**State:** all server-side (the `McpConnection` table) → page server component → client section props. No
new client state beyond display. **Relative-time:** check `src/lib/utils.ts` for an existing formatter; if
none, add a small `formatRelative(date)` ("just now", "3h ago", "5d ago").

**Non-obvious constraints:** use the `"personal"` sentinel for access-token `clientId` so the unique index
keys to one row; the throttle keeps `authenticate()` from hammering the DB on multi-call sessions; the FK
cascades so deleting/soft-deleting a user (R18.2) cleans up their connection rows.

## Tests

- [ ] `pnpm build` clean + `prisma generate` succeeds with the new model/enum.
- [ ] DDL applied to the dev DB (`\d "McpConnection"` shows the table, FK, and unique index) via
      `scripts/apply-schema.ts`.
- [ ] Static token: generate a token → an `ACCESS_TOKEN` connection row appears; make a `tools/list` call →
      `/account` shows "Personal access token · last used just now" with green dot.
- [ ] OAuth: connect via ChatGPT (or replay an OAuth bearer) → an `OAUTH` connection row keyed by the
      client id appears; `/account` lists it with its label + last-used. (Deploy-gated, like R17.2.)
- [ ] Revoke the token → the `ACCESS_TOKEN` row is deleted and it disappears from the list.
- [ ] Upsert throttle: repeated calls within ~5min don't write every time; `lastSeenAt` still advances over
      time.
- [ ] A `/api/mcp` call still succeeds even if the connection upsert is forced to throw (best-effort, no 500).
