# R15.1 — Role Builder strips 4 permissions on any role edit

**Status:** planned
**Files:** `src/app/(app)/pm/users/page.tsx` (the `ALL_PERMISSIONS` checkbox list)

## Spec

**Problem (HIGH):** Saving any role in the Role Builder silently removes the permissions
`MANAGE_CALENDAR`, `VIEW_LEAD_MEETINGS`, `MANAGE_STATUS_UPDATES`, `MANAGE_MEETING_RECORDS`.

Mechanism: `roles.ts` `parsePermissions(formData)` returns
`Object.values(Permission).filter((p) => formData.get(\`perm_${p}\`) === "on")` — i.e. it keeps **only**
permissions whose checkbox was submitted. The Role Builder form in `pm/users/page.tsx` builds its checkboxes
from a **local** `ALL_PERMISSIONS` array that lists only **13 of the 17** `Permission` enum values; the four
added in Sets 10–11 are absent. With no checkbox, those four are never `"on"`, so `updateRole` (and
`createRole`) drop them. Editing the **Project Manager** role removes calendar management, lead-meeting
visibility, status-update management, and meeting-record deletion; editing **Eboard** removes
`VIEW_LEAD_MEETINGS` + `MANAGE_STATUS_UPDATES`. A PM only has to open a role and click "Save role" once.

**Approach:** Add the four missing permissions (with human labels, in the app's own voice) to the page's
`ALL_PERMISSIONS` list so every enum value has a checkbox. Keep the existing order; append the new ones
grouped logically:
```ts
{ value: Permission.MANAGE_CALENDAR,        label: "Manage calendar" },
{ value: Permission.VIEW_LEAD_MEETINGS,     label: "View lead / eboard meetings" },
{ value: Permission.MANAGE_STATUS_UPDATES,  label: "Manage status updates" },
{ value: Permission.MANAGE_MEETING_RECORDS, label: "Manage meeting records" },
```
**Decision (resolved):** keep `roles.ts` as the single source of truth that iterates the full enum — the
bug is the *form* being out of sync, not the parser. To prevent silent regressions when future permissions
are added, the page should derive its checkbox set so a missing label can't drop a permission. Pragmatic,
low-risk version for this set: complete the list now **and** add a dev-time guard — assert that
`ALL_PERMISSIONS` covers every `Object.values(Permission)` (a module-level check that throws in dev if a
new enum value lacks a label), so the next added permission fails loudly instead of being silently
strippable. (Labels stay curated; the guard just guarantees completeness.)
