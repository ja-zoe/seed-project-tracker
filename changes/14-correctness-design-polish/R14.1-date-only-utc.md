# R14.1 — Date-only fields render one day early in negative-UTC timezones

**Status:** tests passing
**Files:** `src/lib/utils.ts` (add helper) + the date-only display sites: `src/app/(app)/my-tasks/page.tsx`,
`src/app/(app)/dashboard/page.tsx`, `src/app/(app)/action-items/page.tsx`,
`src/app/(app)/projects/[id]/page.tsx`, `src/app/(app)/projects/[id]/timeline/page.tsx`,
`src/app/(app)/projects/[id]/history/page.tsx`, `src/app/(app)/pm/review/page.tsx`,
`src/components/sortable-deliverables.tsx` (date formatters)

## Spec

**Problem:** Date-only fields are stored as UTC-midnight `DateTime` (they come from `type="date"` inputs →
`new Date("2026-12-15")` = `2026-12-15T00:00:00.000Z`) but are displayed with `toLocaleDateString(...)`
**without** `timeZone:"UTC"`. For users behind UTC — the entire user base is Rutgers (US Eastern) — that
renders the **previous day**. Reproduced in R13.4: a `2026-12-15` deadline showed "Dec 14" under
`America/New_York`. ~22 `toLocaleDateString` sites; the date-only ones are the bug.

**Approach:** Add a shared formatter and apply it to **date-only** fields only; leave true datetimes alone.

- `src/lib/utils.ts`: add
  ```ts
  export function formatDateOnly(d: Date | string, opts: Intl.DateTimeFormatOptions = { month: "short", day: "numeric" }) {
    return new Date(d).toLocaleDateString("en-US", { ...opts, timeZone: "UTC" });
  }
  ```
  (and reuse it inside `formatProjectDate`/`projectDuration` in utils.ts, which format project start/end —
  also date-only.)

- **Date-only fields → `formatDateOnly`:** `ActionItem.deadline`, `Subtask.dueDate`,
  `Deliverable.targetDate` / `startDate`, `MeetingRecord.meetingDate`, `Project.startDate` / `endDate`.
  Sites: my-tasks (deadline + subtask dueDate), dashboard (deadline; and the review-week label when it
  comes from a meeting record), action-items global list (deadline), project page (meeting-record date),
  timeline (deliverable dates), pm/review (meeting-record date), `sortable-deliverables` date formatters
  (targetDate/dueDate). The action-items **client section** (`action-items-section.tsx`) already uses
  `timeZone:"UTC"` from R13.4 — leave it; this feature brings the rest in line.

- **True datetimes → leave as local** (they carry a real time-of-day): `createdAt`, `completedAt`,
  `updatedAt`, `CalendarEvent.startsAt/endsAt`, and `StatusUpdate.meetingDate` (derived from the lead
  meeting's `startsAt`). In the mixed history timeline, format **meeting-record** entries with
  `formatDateOnly` and **status-update** entries with the existing local format (resolve per entry `kind`).

**Decision (resolved, not an Open Question):** classify by field semantics, not by sniffing `00:00:00Z`,
to avoid a real-midnight datetime being mis-bucketed. The list above is the authoritative classification;
when implementing, confirm each ambiguous site's field source before switching it.
