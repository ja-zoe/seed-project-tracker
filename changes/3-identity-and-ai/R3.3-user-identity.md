# R3.3 — User identity: firstName, lastName, nickname

**Status:** done
**Files:** `prisma/schema.prisma`, `src/lib/utils.ts`, `src/app/profile/setup/`, `src/lib/actions/users.ts`, `src/auth.ts`, `src/app/(app)/layout.tsx` (+ all name-rendering surfaces)

## Spec

**Problem:** `User` only had a single nullable `name` (often a raw netID from CAS). The app showed `user.name ?? user.email`, rendering raw netIDs.

**Goal:** Collect first/last name + optional nickname. Display name = `nickname ?? firstName ?? name ?? email.split("@")[0]`. First/last mandatory beyond the initial pending state.

**DB changes:**
```sql
ALTER TABLE "User" ADD COLUMN IF NOT EXISTS "firstName" TEXT;
ALTER TABLE "User" ADD COLUMN IF NOT EXISTS "lastName"  TEXT;
ALTER TABLE "User" ADD COLUMN IF NOT EXISTS "nickname"  TEXT;
```
`name` retained for NextAuth; a computed display name is derived from the helper.

**Profile gate:** When `firstName` is null, redirect to `/profile/setup` (First/Last required, Nickname optional) before any other route. `updateProfile` sets the three fields and `name = firstName + " " + lastName`. PM not exempt. Gate enforced in the `(app)` layout server component (not middleware).

**Display helper:** `getDisplayName(user)` in `src/lib/utils.ts` → `nickname ?? firstName ?? name ?? email.split("@")[0]`. Used across dashboard greeting, team lists, action item owners, subtask assignees, meeting records, notifications, PM user list.

**Session token:** JWT callback forwards `firstName`/`nickname`.
