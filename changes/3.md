## Feature specs

### R3.1 — Plain-text / Markdown toggle on description fields

**Problem:** The `MarkdownEditor` textarea is the only input mode for descriptions on deliverables, subtasks, and status updates. Some users may want a plain text mode without markdown syntax.

**Scope:** Everywhere `MarkdownEditor` is rendered:
- `src/app/(app)/projects/[id]/deliverables/new/page.tsx`
- `src/app/(app)/projects/[id]/deliverables/[did]/edit/page.tsx`
- `src/app/(app)/projects/[id]/deliverables/[did]/subtasks/new/page.tsx`
- `src/app/(app)/projects/[id]/deliverables/[did]/subtasks/[sid]/edit/page.tsx`
- `src/app/(app)/projects/[id]/status/new/page.tsx` (if it uses MarkdownEditor)
- `src/app/(app)/projects/[id]/meeting/new/page.tsx` (if applicable)

**Approach:**
- Add a small `MD / Plain` pill toggle above the textarea (top-right corner of the label row).
- `MarkdownEditor` gains an optional `allowToggle` prop (default `true`).
- The toggle is local state in the component — no DB storage of mode preference needed (the stored value is always the raw text; the mode only affects whether markdown syntax hints / GFM rendering previews are shown).
- In `MD` mode: current behavior (monospace textarea, placeholder shows markdown syntax).
- In `Plain` mode: proportional font textarea, simplified placeholder, no markdown hints. The field name and stored value are identical — just the presentation changes.
- Optionally: in `MD` mode, add a split-pane preview toggle (edit | split | preview) using the existing `react-markdown` + `remark-gfm` already installed. This is a stretch goal and should be called out as optional in implementation.

**No DB changes needed.**

### R3.2 — Inline subtask delete with slide-in confirmation

**Problem:** The `SortableDeliverables` client component (`src/components/sortable-deliverables.tsx`) shows an "Edit" link per subtask but has no delete capability. Per-deliverable delete exists at the deliverable level but subtask delete is absent from the UI entirely.

**Desired UX:**
- Each subtask row shows: `[dot] [title] [assignee] ... [date] [Edit]`
- To the right of "Edit", add a delete trigger — an `X` icon (`XCircle` from Phosphor, Bold weight) in muted color.
- When clicked, the trigger initiates a confirmation micro-interaction:
  - The "Edit" and "X" controls slide out to the left (CSS `translateX` + `opacity` transition, ~200ms ease-out).
  - Simultaneously, "Yes / No" slide in from the right edge of the subtask row.
  - "Yes" styled in `text-[#A4503C]` (clay/destructive). "No" styled in muted.
  - Clicking "No" reverses the animation and restores "Edit" + "X".
  - Clicking "Yes" calls a `deleteSubtask` server action (new) and removes the row with a fade-out.
- This is entirely client-side state within `SortableDeliverables` — one `confirmingDelete: string | null` state (holds the subtask id being confirmed, or null).
- The animation should use CSS transitions (Tailwind `transition-all duration-200`) rather than GSAP to keep it lightweight.

**New server action needed:** `deleteSubtask(subtaskId: string)` in `src/lib/actions/deliverables.ts`
- Auth: project membership (any role) or `MANAGE_MILESTONES` permission.
- Deletes the subtask and revalidates the project path.

**No DB changes needed.**

### R3.3 — User identity: firstName, lastName, nickname

**Problem:** The `User` model only has a single nullable `name String?` field populated from CAS (typically a netID or full name from LDAP). The app currently displays `user.name ?? user.email` in greetings and throughout the UI, which can render as a raw netID.

**Goal:** Collect first name, last name, and optional nickname. Use `nickname` as the display name if set, otherwise `firstName`. `firstName` and `lastName` are mandatory for any user beyond initial pending state.

**DB changes:**
```sql
ALTER TABLE "User" ADD COLUMN IF NOT EXISTS "firstName" TEXT;
ALTER TABLE "User" ADD COLUMN IF NOT EXISTS "lastName"  TEXT;
ALTER TABLE "User" ADD COLUMN IF NOT EXISTS "nickname"  TEXT;
```
Applied via `scripts/apply-schema.ts`.
`prisma/schema.prisma` additions to `User` model:
```
firstName String?
lastName  String?
nickname  String?
```
The existing `name` field is retained (used by NextAuth adapters) but is now considered the "raw/legacy" name. A computed display name is derived from `nickname ?? firstName ?? name ?? email.split("@")[0]`.

**Profile completion gate:**
- After first login (when `firstName` is null), redirect users to `/profile/setup` before they can access any other route.
- `/profile/setup` is a simple form: First Name (required), Last Name (required), Nickname (optional).
- On submit: `updateProfile` server action sets `firstName`, `lastName`, `nickname`, and also sets `name = firstName + " " + lastName` for NextAuth compatibility.
- The PM (auto-activated) is not exempt — they must complete profile too.
- Middleware or the app layout guards the redirect: if `session.user.firstName` is null → redirect to `/profile/setup`.

**Display name helper:**
- Add `getDisplayName(user: { firstName?: string | null; nickname?: string | null; name?: string | null; email: string }): string` to `src/lib/utils.ts` (or a new `src/lib/display-name.ts`).
- Returns: `nickname ?? firstName ?? name ?? email.split("@")[0]`.

**App-wide changes:**
- Dashboard greeting: "Good to see you, [displayName]." uses `getDisplayName`.
- All places that render `user.name ?? user.email.split("@")[0]` (project team lists, action item owners, subtask assignees, meeting records, notification recipients, PM user list) switch to `getDisplayName(user)`.
- Supporting/secondary text continues to show email or role/title where relevant (e.g., smaller muted text under a name in the team list).
- `src/app/(app)/pm/users/page.tsx` — user list shows full name as primary, email as secondary.

**Session token:** The JWT session callback in `src/auth.ts` needs to forward `firstName` and `nickname` so `getDisplayName` can be called client-side from the session without a DB lookup. Or alternatively, `getDisplayName` is always called server-side. Either pattern is acceptable; keep it server-side for simplicity.

### R3.4 — AI agent chatbot (bring-your-own-AI)

**Goal:** A role-scoped AI assistant that can autonomously perform work within the dashboard on behalf of the logged-in user — creating action items, drafting status updates, building timelines, assigning subtasks — all constrained by that user's actual permissions. Exposed as a floating chat window at the bottom-right of every app page.

**Cost model:** The AI cost burden falls entirely on the individual user. No platform-level AI API keys are held by the app. Users connect whichever providers they have accounts with — via OAuth where the provider supports it, or via API key otherwise. A user can connect multiple providers and switch between them and their respective models in a single dropdown in the chat header.

---

#### Protocol standardization: MCP as the tool-calling layer

All tool use (the SEED dashboard actions the AI can take) is exposed to the model via the **Model Context Protocol (MCP)**. MCP is a provider-agnostic, open standard for how models discover and call tools — every major provider (Claude, OpenAI, Gemini, Copilot, DeepSeek) either already supports it or will. The SEED tools are defined once as an MCP server; the Vercel AI SDK's `experimental_createMCPClient` connects each provider to that same server. This means:

- Adding a new provider in the future requires no changes to tool definitions — only the auth/connection layer changes.
- Switching models mid-conversation is safe: the tool schema the model receives is identical regardless of which model is active.
- A user's own external MCP server (e.g. their company's MCP endpoint) can be layered in alongside the SEED tools.

The Vercel AI SDK (`ai` package) handles the transport: it implements both the MCP client (connecting to SEED's tool server) and the model-calling interface (`streamText`), making the whole stack uniform regardless of provider.

---

#### Provider support and connection methods

| Provider | Auth method | SDK package | Notes |
|---|---|---|---|
| Claude (Anthropic) | API key *(OAuth roadmap)* | `@ai-sdk/anthropic` | Key from console.anthropic.com. Anthropic does not currently offer OAuth for API access; UI should be designed to accept OAuth in a future update without layout changes. |
| ChatGPT (OpenAI) | API key *(OAuth roadmap)* | `@ai-sdk/openai` | Key from platform.openai.com. Same situation as Claude — no public OAuth for API access today. |
| Gemini (Google) | **Google OAuth** or API key | `@ai-sdk/google` | OAuth preferred — user signs in with Google; app exchanges token for Gemini API access via Google AI Studio. API key as fallback. |
| GitHub Copilot | **GitHub OAuth** or API key | `@ai-sdk/openai` (OpenAI-compat) | OAuth via GitHub grants Copilot access. OpenAI-compatible API; uses `baseURL` override. |
| DeepSeek | API key *(OAuth roadmap)* | `@ai-sdk/openai` (OpenAI-compat) | OpenAI-compatible endpoint. No OAuth today. |
| Custom MCP server | Endpoint URL (+ optional bearer token) | `ai` `experimental_createMCPClient` | User supplies their own MCP server URL. Layers additional tools alongside the SEED tool server. |

**Design principle:** The connection UI card for every provider shows both an "OAuth / Sign in" button and an "API key" input, regardless of current support. For providers that don't support OAuth yet, the OAuth button is disabled with a tooltip: "OAuth coming soon — use an API key for now." This means no UI rework is needed when Anthropic or OpenAI eventually ship OAuth.

**OAuth flow (for Gemini and Copilot):**
- A "Sign in with Google" / "Sign in with GitHub" button triggers a provider-specific OAuth popup or redirect.
- On return, the access token is stored encrypted in `UserAiProvider` (same DB model as API keys). It is refreshed automatically before expiry.
- Next.js already uses Auth.js v5 for CAS auth; the OAuth connections for AI providers are separate — they use Auth.js's `signIn("google")` / `signIn("github")` flows but are scoped only to AI access (not app login). The Google/GitHub OAuth providers need to be added to the Auth.js config but marked as `linkOnly: true` (not used for app session auth).
- OAuth connections appear in the chat settings panel with a "Connected as [account name]" indicator and a Disconnect button.

#### Backend — streaming API route

`src/app/api/chat/route.ts`
- POST; requires session.
- Accepts:
  ```ts
  {
    provider: "anthropic" | "openai" | "google" | "copilot" | "deepseek" | "mcp";
    credential: string;   // API key, OAuth access token, or MCP endpoint URL
    model: string;        // fully-qualified model string e.g. "claude-haiku-4-5-20251001"
    messages: CoreMessage[];
    projectId?: string;
  }
  ```
- Credential is used only within this request and never written to any log.
- Builds system prompt: user role, permissions list, current project context.
- Calls `src/lib/ai-provider.ts#resolveModel(provider, credential, model)` to get the Vercel AI SDK model instance.
- Connects to the SEED MCP tool server (running in-process or as a separate Next.js route) via `experimental_createMCPClient`.
- Calls `streamText({ model, system, messages, tools: mcpTools, maxSteps: 5 })`.
- For custom MCP provider: merges tools from the user's MCP endpoint with SEED's own tools.

**SEED MCP tool server:** `src/app/api/mcp/route.ts`
- A Next.js route that implements the MCP SSE protocol.
- Exposes the same tool set regardless of which AI model is calling it:

| Tool | Description | Permission required |
|---|---|---|
| `createActionItem` | Create an action item on a project | `ASSIGN_ACTION_ITEMS` or LEAD/SUBLEAD |
| `createStatusUpdate` | Draft and submit a status update | `POST_MEETING_TRACKING` or LEAD/SUBLEAD |
| `listProjects` | Return user's visible projects | any authenticated user |
| `getProjectDetail` | Deliverables + open action items for a project | membership or `VIEW_ALL_PROJECTS` |
| `createDeliverable` | Create a deliverable | `MANAGE_MILESTONES` |
| `createSubtask` | Create a subtask on a deliverable | membership or `MANAGE_MILESTONES` |
| `updateProjectStatus` | Change a project's status | LEAD/SUBLEAD or `MANAGE_PROJECTS` |

Every tool validates the session's actual permissions before executing.

**Default model per provider** (shown in UI; user can override):

| Provider | Default |
|---|---|
| Claude | `claude-haiku-4-5-20251001` |
| OpenAI / Copilot | `gpt-4o-mini` |
| Gemini | `gemini-2.0-flash` |
| DeepSeek | `deepseek-chat` |

#### Encrypted credential storage (DB)

New `UserAiProvider` model — one row **per provider per user** (not unique on userId alone, since a user can connect multiple):

```prisma
model UserAiProvider {
  id               String   @id @default(cuid())
  userId           String
  provider         String   // "anthropic" | "openai" | "google" | "copilot" | "deepseek" | "mcp"
  encryptedCred    String   // AES-256-GCM encrypted API key, access token, or MCP URL
  selectedModel    String?  // last-used model for this provider
  accountLabel     String?  // e.g. "user@gmail.com" for OAuth providers
  createdAt        DateTime @default(now())
  updatedAt        DateTime @updatedAt

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([userId, provider])
}
```

API endpoints:
- `GET /api/user/ai-providers` — returns list of connected providers with `{ provider, selectedModel, accountLabel, hasCredential: true }` (never returns raw credential).
- `PUT /api/user/ai-providers/[provider]` — upserts the encrypted credential + model preference.
- `DELETE /api/user/ai-providers/[provider]` — disconnects a provider.

Encryption helper `src/lib/crypto.ts`:
```ts
encrypt(plaintext: string, secret: string): string
decrypt(ciphertext: string, secret: string): string
```

Required env var:
```
AI_KEY_ENCRYPTION_SECRET=<32-byte random hex>
```
If absent, the "Save to my account" checkbox is hidden and session-only `sessionStorage` is the only option.

#### Security notes

- OAuth tokens and API keys travel browser → Next.js route → AI provider over HTTPS only. Never logged.
- The route checks `requireAuth()` before reading any credential from the request body.
- MCP server URLs must be HTTPS; the route rejects `http://` URLs.
- `AI_KEY_ENCRYPTION_SECRET` rotation invalidates all saved credentials (users must reconnect). Document this in the admin runbook.
- Google/GitHub OAuth connections for AI are separate from app authentication. A user disconnecting their Google AI connection does not affect their app login.

## Open questions / decisions before implementing

1. **R3.1 plain-text toggle persistence:** Should the selected mode (MD vs Plain) be remembered per-form-visit in `localStorage`, or always default to MD? Recommendation: always default to MD; no persistence needed.

2. **R3.3 profile gate:** Should the gate redirect from middleware or from inside the layout server component? Recommendation: layout server component check (simpler, already has session). Middleware would need to forward session fields.

3. **R3.3 retroactive names:** Existing users (seeded or from CAS) will have null `firstName`/`lastName`. They'll be gated on first visit after the migration. The PM is not exempt. Acceptable.

4. **R3.4 MCP tool server:** The SEED MCP server is proposed as a Next.js API route (`/api/mcp`) using SSE. An alternative is running it in-process using the Vercel AI SDK's in-memory MCP adapter (no separate route needed). In-process is simpler for Rev 3. Confirm approach before implementing.

5. **R3.4 encrypted credential storage:** Ship DB-backed encrypted storage in Rev 3 (requires `AI_KEY_ENCRYPTION_SECRET` env var + `UserAiProvider` DDL), or defer to Rev 4 and use session-only `sessionStorage` first? Recommendation: ship session-only first; opt-in save in Rev 4. Confirm.

6. **R3.4 OAuth providers (Gemini, Copilot):** OAuth for AI account linking requires registering separate OAuth apps (Google AI Studio, GitHub). This is straightforward but needs credentials added to `.env`. Confirm whether OAuth sign-in should be in Rev 3 or if API key only is sufficient for the initial build.

7. **R3.4 custom MCP server:** User-supplied MCP endpoint is a power-user feature. `experimental_createMCPClient` is still marked experimental in the Vercel AI SDK. Recommendation: defer custom MCP to Rev 4; ship the five named providers first. Confirm.

8. **R3.4 streaming:** `useChat` + `streamText` is the recommendation. Confirm streaming is desired.

9. **R3.4 chat history persistence:** Client memory only in Rev 3 (lost on reload). `ChatSession` DB model can be added in Rev 4. Confirm acceptable.
