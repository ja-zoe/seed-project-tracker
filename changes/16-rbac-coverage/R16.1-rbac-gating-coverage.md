# R16.1 — RBAC gating regression coverage

**Status:** tests passing
**Files:** `e2e/r16-rbac-audit.spec.ts` (new). No source change.

## Spec

**Problem:** The app's access gates — the PENDING wall and global-role page-gating — are security-critical
and had **zero automated coverage**. Set 15 just fixed a *silent* RBAC regression (the Role Builder
stripping permissions), which is exactly the class of bug that slips through without tests. A regression
guard for the gates is warranted.

**Approach:** A multi-context Playwright spec (`e2e/r16-rbac-audit.spec.ts`) that exercises the real flow:
1. A fresh dev-login netId is created `PENDING` → asserts the page lands on `/pending` with the
   "Awaiting approval" heading (screenshot saved).
2. Activate that user as a **Viewer** deterministically via Prisma (`status: ACTIVE`, the Viewer `roleId`,
   and `firstName` — without `firstName` the app layout diverts to `/profile/setup`). Setup goes through
   the DB rather than driving the multi-user approval UI, so the test is deterministic regardless of how
   many other users exist.
3. Re-login → asserts `/dashboard` (active user, past the PENDING + profile gates).
4. The Viewer visits `/pm/users`, `/pm/settings`, `/pm/review` directly → asserts each redirects to
   `/dashboard` (`requirePermission` denies `MANAGE_USERS` / `CONFIGURE_NOTIFICATIONS` / `VIEW_MONTHLY_REVIEW`).

**Hygiene (decided):** the test owns its data — it creates one throwaway user `auditv<ts>@scarletmail…` and
deletes it in a `finally` (the global-teardown only handles projects/events, not users). DB access in the
spec uses the same adapter pattern as `scripts/` with `import "dotenv/config"`.

**Why not also the approval UI / project-role layer here:** the approval UI is fragile to ambient user
count and is better covered by a dedicated PM-users spec; the project-role (LEAD/SUBLEAD/MEMBER) write
gating is a separate concern. Scoped this set to the global gates that the audit actually exercised. (A
project-role-gating guard is a clean follow-up — left for a future set, not blocking.)
