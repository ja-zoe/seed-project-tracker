# R6.1 — Email alert to user-managers on new signup

**Status:** in progress
**Files:** `src/auth.ts`, `src/lib/notifications.ts` (or new `src/lib/signup-notify.ts`), `prisma/schema.prisma`, DB enum

## Spec

**Problem:** When a brand-new user signs in via CAS for the first time they are created as
`PENDING` (`src/auth.ts` `authorize()`), but **nobody is told.** A PENDING user sits on
`/pending` indefinitely until a PM happens to open `/pm/users`. People who hold the
`MANAGE_USERS` permission should be proactively notified — by email and in-app — so they can
approve the account promptly.

**Approach:**

1. **Recipients.** All `ACTIVE` users whose role grants `MANAGE_USERS`. Mirror the existing
   `getPMUserIds()` helper in `src/lib/notifications.ts` (which filters on
   `role.permissions.has("MANAGE_PROJECTS")`) but filter on `MANAGE_USERS`. Add an exported
   helper `getUserManagerIds()`.

2. **Trigger point.** In `src/auth.ts` `authorize()`, in the branch that creates a brand-new
   user (`if (!user) { … prisma.user.create(...) }`), **only when the created user is PENDING**
   (i.e. not the auto-promoted PM), fire the notification *after* the user row is created.

3. **Delivery.** Add an exported `notifyNewSignup(newUser: { id; name; email })` to
   `src/lib/notifications.ts`. It builds one `NotificationPayload` per recipient
   (`type: USER_SIGNUP`, `title: "New account awaiting approval"`,
   `body: "<name> (<email>) signed in and is awaiting approval."`, `link: "/pm/users"`) and
   calls the existing `deliver(Channel.BOTH, payloads)` path so it goes to both in-app and
   email (reusing the module's Resend wiring — no new email code). `sendEmail` already no-ops
   when `RESEND_API_KEY` is unset, so mock/dev environments degrade gracefully.

4. **Must not break sign-in.** `authorize()` runs inside the Credentials provider; an email
   failure must never reject the login. Wrap the call: `try { await notifyNewSignup(user); }
   catch (e) { console.error("signup notify failed", e); }`. Do not `await` it in a way that
   propagates throws.

5. **DB change (enum).** Add `USER_SIGNUP` to the `NotificationType` enum (pending Open
   Question 4 — falls back to reusing `GENERAL` if rejected):
   `ALTER TYPE "NotificationType" ADD VALUE IF NOT EXISTS 'USER_SIGNUP';` then add it to
   `prisma/schema.prisma`, `pnpm prisma generate`, restart dev server.

**Edge cases:**
- A returning PENDING user signing in again must **not** re-notify (only the `!user` create
  branch fires — re-logins skip it).
- If there are zero `MANAGE_USERS` holders (fresh DB before PM exists), `notifyNewSignup`
  produces an empty payload list; `deliver` already no-ops on empty input.
