## Spec

**Problem (consolidated from 3 user-reported issues):**

1. **Deliverable status cascade is broken** — `updateSubtaskStatus` previously never touched the
   parent deliverable's status. A fix was added but the derivation rule needs to be authoritative:
   "if any subtask is not NOT_STARTED, the deliverable must reflect that." UI never locks the
   deliverable badge or the edit-form status field, so a PM can manually override a status that
   should be derived from subtasks.

2. **Inline editing is incomplete** — only title and status-via-dot exist. There is no inline
   assignee picker, no inline due date, and the title edit saves on blur/Enter with no confirm
   step — easy to accidentally commit. The delete button already uses a slide-in confirm; all
   inline edits should follow the same pattern.

3. **Standalone-deliverable status is uneditable** — a deliverable with no subtasks cannot have
   its status changed without navigating to the edit form. It should have an inline status popover
   the same way subtasks do.

---

### 1 · Deliverable status derivation rule

Run this in `updateSubtaskStatus` after every subtask update (already partially in place):

```
siblings = all subtasks of the deliverable (AFTER the update)
if ALL COMPLETE          → derivedStatus = COMPLETE
else if ANY BLOCKED      → derivedStatus = BLOCKED
else if ANY IN_PROGRESS or COMPLETE  → derivedStatus = IN_PROGRESS
else                     → derivedStatus = NOT_STARTED
```

`prisma.deliverable.update` sets `status = derivedStatus`, `completed = (derived === COMPLETE)`,
`completedDate = (derived === COMPLETE ? new Date() : null)`.

When the deliverable has **zero subtasks**, do not call the cascade — there are no siblings to
derive from. The deliverable status is manual-only in that case.

### 2 · Deliverable status lock in UI

**`SortableDeliverables` — deliverable badge:**

- **Has subtasks:** render a read-only `<span>` badge + `LockSimple` (Phosphor, size 9) inside a
  `group/badge` wrapper. On hover, show a tooltip div listing each subtask whose status ≠
  NOT_STARTED as `"<title> — <STATUS_LABELS[status]>"`. Use `group-hover/badge:opacity-100`
  opacity transition (starts at 0). Tooltip positioned `bottom-full mb-1`, z-50, dark bg
  (`bg-foreground text-background`), bounded width.
- **No subtasks + canEdit:** render a `<button>` that opens `DeliverableStatusPopover` (same
  design as `StatusPopover` but calls `updateDeliverableStatus(deliverableId, status)`). Track
  open state with `deliverableStatusMenuFor: string | null`.
- **No subtasks + !canEdit:** plain `<span>` badge (no lock icon, no tooltip needed).

**Deliverable edit page (`/deliverables/[did]/edit`):**

Load with `include: { _count: { select: { subtasks: true } } }`. When `_count.subtasks > 0`:

```tsx
{/* Status locked — derived from subtasks */}
<div>
  <label …>Status</label>
  <p className="text-sm text-muted-foreground px-3 py-2.5 rounded-md border border-border/50 bg-muted/40"
     style={{ fontFamily: "var(--font-mono)" }}>
    {STATUS_LABELS[deliverable.status]} — auto-derived from subtasks
  </p>
</div>
```

Omit the `<select>` entirely so no `status` key is submitted. In `updateDeliverable`:

```ts
const statusRaw = formData.get("status") as string | null;
if (statusRaw) {
  // only update status for standalone deliverables (form omits it when locked)
  data.status = statusRaw as TimelineStatus;
  data.completed = statusRaw === "COMPLETE";
  data.completedDate = statusRaw === "COMPLETE" ? new Date() : null;
}
```

### 3 · Inline subtask editing — unified pending-confirm pattern

**State in `SortableDeliverables`:**

```ts
const [pendingEdit, setPendingEdit] = useState<{
  subtaskId: string;
  field: "title" | "assignee" | "dueDate";
  value: string;  // pending new value: text | userId | "YYYY-MM-DD" | ""
} | null>(null);
const [isPendingEdit, startEditTransition] = useTransition();
```

**Right-side controls — 3 panel states per subtask row** (using same `opacity + translate`
slide animation as the existing delete confirm):

```
rowMode = pendingEdit?.subtaskId === id ? "edit"
        : confirmingDelete === id      ? "delete"
        : "controls"

Panel A (controls): [pencil | calendar | person | ×]   — visible when rowMode === "controls"
Panel B (edit):     [✓ CheckFat | ✗ XCircle]          — visible when rowMode === "edit"
Panel C (delete):   [Yes | No]                         — visible when rowMode === "delete"
```

All three panels share a `relative` wrapper; B and C are `absolute right-0`. The inactive
panel uses `opacity-0 pointer-events-none` + translate offset.

Icons for Panel A: `PencilSimple` (title), `CalendarBlank` (dueDate), `UserCircle` (assignee),
`XCircle` (delete). All size 12–13, `text-muted-foreground hover:text-foreground`.

Icons for Panel B: `CheckFat` (size 13, weight "fill", `text-[#588157]` when not pending,
`opacity-50` when `isPendingEdit`) and `XCircle` (size 13, weight "bold", `text-muted-foreground`).

**Entering edit mode** (start any field edit):

```ts
function startEdit(subtaskId: string, field: "title"|"assignee"|"dueDate", value: string) {
  setConfirmingDelete(null);
  setStatusMenuFor(null);
  setPendingEdit({ subtaskId, field, value });
  // focus input on next tick (title and dueDate) — assignee search focuses itself
}
```

**Committing:**

```ts
function commitEdit() {
  if (!pendingEdit) return;
  const { subtaskId, field, value } = pendingEdit;
  setPendingEdit(null);
  startEditTransition(async () => {
    if (field === "title") await updateSubtaskTitle(subtaskId, value);
    else if (field === "assignee") await updateSubtaskAssignee(subtaskId, value || null);
    else if (field === "dueDate") await updateSubtaskDueDate(subtaskId, value || null);
  });
}
```

**Title edit (left side):** when `pendingEdit?.field === "title"`, replace the title `<span>`
with an `<input>` (no onBlur save — wait for ✓). Changes update `pendingEdit.value`.

**Due date edit (right side):** when `pendingEdit?.field === "dueDate"`, replace the date
display chip with a compact `<input type="date">`. Changes update `pendingEdit.value`.

**Assignee picker (left side):** when `pendingEdit?.field === "assignee"`, show
`<AssigneeSearch>` — a `relative` wrapper containing:
- A small `<input>` for search (monospaced, autofocused)
- An absolute drop-down list of all `members` filtered by search string (case-insensitive
  match on `getDisplayName`), capped at `max-h-40 overflow-y-auto`
- A "None" row at the top (sets value to `""`, clears assignee)
- Each row shows a `UserCircle` icon + display name; clicking sets `pendingEdit.value = member.id`
  but does NOT commit — that happens on ✓ click.

Close the picker on outside-click or Escape using a `useEffect` listener (same pattern as
`StatusPopover`).

**Optimistic display while pending:** show the pending assignee (look up
`members.find(m => m.id === pendingEdit.value)`) instead of `subtask.assignee`, and show the
pending due-date value instead of `subtask.dueDate`.

### 4 · Status-dot popover (subtask) — unchanged

The existing `StatusPopover` and dot-button remain. Selecting a status immediately commits
(no confirm step — single-tap is the confirm). This is the one intentional exception to
the slide-in rule, because the popover itself is a confirm surface.

### 5 · New server actions

All in `src/lib/actions/deliverables.ts`, same auth pattern as existing subtask actions:

```ts
updateDeliverableStatus(deliverableId, status)
  → requires no subtasks (throws if _count.subtasks > 0)
  → requires project membership or MANAGE_MILESTONES
  → sets completed/completedDate appropriately
  → revalidatePath

updateSubtaskAssignee(subtaskId, assigneeId | null)
  → requires membership or MANAGE_MILESTONES
  → sets assigneeId (null to unassign)
  → revalidatePath

updateSubtaskDueDate(subtaskId, dueDate | null)
  → requires membership or MANAGE_MILESTONES
  → parses dueDate string to Date (or null to clear)
  → revalidatePath
```

### 6 · Props change on SortableDeliverables

Add `members: Member[]` where `Member = { id, firstName, nickname, name, email }`.
Page passes `members={project.assignments.map(a => a.user)}`.

### Edge cases

- Clicking a different subtask's edit icon while `pendingEdit` is open for another subtask:
  `startEdit` clears the old pendingEdit first (no double-confirm).
- `commitEdit` called with empty title string: `updateSubtaskTitle` already rejects; the
  action throws, Next.js surfaces an error toast; no crash.
- Deliverable with no subtasks, user sets status to COMPLETE: `updateDeliverableStatus` sets
  `completed = true`, `completedDate = now()`.
- Picker outside-click on the assignee row itself (e.g. clicking the "None" option): the
  `handleOutside` check excludes the anchor element.
