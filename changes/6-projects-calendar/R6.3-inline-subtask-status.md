# R6.3 ‚Äî Inline subtask status change

**Status:** planned
**Files:** `src/components/sortable-deliverables.tsx`

## Spec

**Problem:** On the project page, a subtask's status is shown as a small colored dot
(`STATUS_DOT[subtask.status]` in `sortable-deliverables.tsx`) but is read-only ‚Äî to change
`NOT_STARTED ‚Üí IN_PROGRESS ‚Üí COMPLETE/BLOCKED` you must open the full subtask edit form
(`/deliverables/[did]/subtasks/[sid]/edit`). That's heavy for a one-field change members make
constantly. The server action already exists.

**Approach:**

- The server action `updateSubtaskStatus(subtaskId, status)` already exists in
  `src/lib/actions/deliverables.ts` and is correctly gated (project membership, else
  `MANAGE_MILESTONES`) and already `revalidatePath`s the project. **No server changes needed** ‚Äî
  this is wiring only.
- `sortable-deliverables.tsx` is already a client component (it uses `useState` for
  `confirmingDelete`). Turn the existing status dot into a button that opens a small **status
  popover** (resolved: dropdown, not click-to-cycle): the 4 `TimelineStatus`
  values, each as a row with its colored dot (`STATUS_DOT`) + label, with the current status
  marked. Selecting one calls `updateSubtaskStatus(subtask.id, status)` and closes the popover;
  `revalidatePath` from the action refreshes the list (wrap in a `startTransition` so the row
  shows a pending state and doesn't appear frozen).
- Track open state with `statusMenuFor: string | null` (subtask id), mirroring the existing
  `confirmingDelete` pattern. Close on outside-click / Escape. Position the popover relative to
  the dot; keep it inside the viewport (the notification-modal clipping bug from R5.1 is the
  cautionary tale ‚Äî avoid negative left offsets).
- Permission affordance: only render the dot as an interactive button when the viewer can edit.
  The component receives the deliverables from the page; pass a `canEdit: boolean` prop
  (computed on the page from membership/`MANAGE_MILESTONES`, same logic the page already uses to
  show add/delete affordances) and fall back to a static dot when `false`.

**Edge cases:**
- Setting status to `COMPLETE` sets `completedAt` (the action already handles this); the dot
  color updates on revalidate.
- Rapid re-clicks while a transition is pending: disable the popover rows during
  `isPending`.

## Tests
