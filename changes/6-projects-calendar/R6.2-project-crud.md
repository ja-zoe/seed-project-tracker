# R6.2 — Project create / update / delete (app + MCP)

**Status:** in progress
**Files:** `src/lib/actions/projects.ts`, new `src/app/(app)/projects/[id]/edit/page.tsx`,
`src/app/(app)/projects/[id]/page.tsx` (delete button + edit link), `src/app/api/mcp/route.ts`,
`src/app/(app)/account/page.tsx` (tools list)

**Depends on R6.4** for the `startDate`/`endDate` form inputs — implement R6.4's schema first,
then this feature's edit form includes those two date fields. Both features edit
`projects.ts`, the edit page, and the MCP create/update tools, so **do not parallelize R6.2
and R6.4** — one agent owns both, R6.4 first.

## Spec

**Problem:** There is no way to rename a project or edit its description after creation.
`page.tsx:179` links to `/projects/${id}/edit`, but **that page does not exist**, and
`updateProject()` in `projects.ts` is a no-op on auth (`void user; // permission checked in
page` — but no page enforces it). There is also no way to delete a project. MANAGE_PROJECTS
holders need full create/update/delete in the app, and create/update over MCP.

**Approach:**

1. **Harden `updateProject`.** Replace the `requireAuth()` + `void user` stub with
   `await requirePermission(Permission.MANAGE_PROJECTS);` at the top (matching `createProject`).
   Keep the existing field handling (name, description, semester, correctiveActionPlan) and add
   `startDate`/`endDate` parsing from R6.4. Keep `revalidatePath` calls.

2. **New `deleteProject(projectId)` server action** in `projects.ts`:
   `await requirePermission(Permission.MANAGE_PROJECTS);` then
   `await prisma.project.delete({ where: { id: projectId } });`. All `Project` relations
   (`assignments`, `deliverables`, `statusUpdates`, `meetingRecords`, `actionItems`) are
   `onDelete: Cascade` in the schema, so this is safe. `revalidatePath("/projects")` then
   `redirect("/projects")`.

3. **Edit page** `src/app/(app)/projects/[id]/edit/page.tsx` (server component):
   - Guard with `await requirePermission(Permission.MANAGE_PROJECTS)` (redirects non-holders
     to /dashboard).
   - Load the project, render a form bound to `updateProject.bind(null, id)` with inputs for
     name, semester, description (textarea), startDate (`type="date"`), endDate (`type="date"`,
     optional), correctiveActionPlan (textarea). Reuse the styling of the existing
     `/projects/[id]/deliverables/[did]/edit` form for visual consistency.
   - Below the form, a **Delete project** danger zone: a small client component with a
     two-step confirm (button → "Are you sure? This permanently deletes all deliverables,
     meetings, and action items" → confirm) that calls `deleteProject.bind(null, id)`. Reuse
     the inline-confirm pattern from `sortable-deliverables.tsx` (`confirming` boolean state,
     `XCircle`/`Trash` Phosphor icons).

4. **Surface the edit entry point** on `page.tsx`: show an "Edit project" link/button in the
   project header, visible only when the viewer has MANAGE_PROJECTS (the page already computes
   permissions). Fix the existing dangling `/projects/${id}/edit` reference at line ~179 to
   point at the real page.

5. **MCP tools** in `src/app/api/mcp/route.ts` — add two tools to the `TOOLS` array and two
   `executeTool` cases, **gated by `permissions.includes(Permission.MANAGE_PROJECTS)`**
   (return `{ error: "Requires MANAGE_PROJECTS permission" }` otherwise — same shape as other
   permission errors in the file). **No delete tool** (per requirement, MCP offers create +
   update only):
   - `create_project` — args: `name` (req), `semester` (req), `description?`, `startDate?`
     (ISO), `endDate?` (ISO). Creates and returns `{ id, name }`.
   - `update_project` — args: `projectId` (req) + any of `name`, `semester`, `description`,
     `startDate`, `endDate`, `correctiveActionPlan`. Partial update (only set provided fields).
     Returns the updated `{ id, name, ... }`.
   - Bump `serverInfo.version` (currently `2.0.0`) to reflect the new tool count.

6. **Account page** (`/account`) — extend the documented tools list (last touched in R5.5) from
   14 to 16 tools, adding `create_project`/`update_project` under a "MANAGE_PROJECTS" RBAC note.

**Edge cases:**
- `update_project` with no updatable fields → return `{ error: "No fields to update" }`.
- Deleting the project you're currently viewing → the action redirects to `/projects`, so the
  now-404 page is never re-rendered.
