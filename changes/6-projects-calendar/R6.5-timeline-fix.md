# R6.5 — Fix deliverable timeline visual artifacts

**Status:** tests passing
**Files:** `src/app/(app)/projects/[id]/timeline/page.tsx` (and possibly extract a small
client/helper component for the track)

## Spec

**Problem:** The "Deliverable Timeline" Gantt on `/projects/[id]/timeline` is the right idea
(chronological bars per deliverable with subtask bars beneath) but renders with visible
artifacts. Concretely, in the current implementation:

1. **Today line misalignment.** The date-axis row uses `ml-[220px]` and positions the "today"
   line at `left: nowPct%` within that margin-offset container. Each deliverable row instead
   lays out as `[220px label][flex-1 track][w-24 date]`, so its own today line (also
   `left: nowPct%`) is measured against a *different* horizontal extent (it excludes the 24px
   trailing date column the axis doesn't account for, and the axis excludes the 220px via
   margin while rows use a flex child). Result: the axis "today" marker and the per-row today
   lines don't line up vertically.
2. **Null-start bars span the whole width.** A deliverable with `startDate == null` uses
   `startPct = 0`, drawing a bar from the far left to its target — visually implying it started
   at the very beginning of the range, which is wrong/misleading.
3. **Negative offsets on subtasks.** A subtask with no `startDate` uses
   `left: pct(dueDate) - 5%`, which goes **negative** when the due date is near `minDate`,
   overflowing the track to the left.
4. **Degenerate range.** With one date (or all-equal dates), `rangeMs = max(…, 1)` collapses
   everything to ~100%, bunching all bars at the right edge.

**Approach:**

1. **Unify the horizontal track.** Restructure axis + rows to a shared CSS grid:
   `grid-template-columns: 220px 1fr 96px` (label · track · date), used identically by the
   axis row and every deliverable/subtask row. Render the **today line and the min/max axis
   ticks as a single absolutely-positioned overlay over the middle (track) column only**, so
   there is one source of truth for `x = nowPct%` and everything aligns. The track column is
   the only place percentages are computed against.
2. **Milestone vs. bar.** When a deliverable has no `startDate`, render a **diamond/marker at
   `targetPct`** (a milestone) instead of a full-width bar. Only render a bar when both a
   start and a target exist and `start < target`. Same rule for subtasks: no `startDate` ⇒
   small dot marker at `duePct`, never a synthetic `-5%` bar.
3. **Clamp.** `pct()` already clamps to `[0,100]`; ensure every `left`/`right` derives from it
   (no raw `pct(x) - 5`). Replace the subtask `minWidth:4px` hack with the marker approach.
4. **Pad degenerate ranges.** If `maxDate - minDate` is below a floor (e.g. < 7 days, or only
   one distinct date), pad `minDate`/`maxDate` to a sensible window (e.g. start − 1wk,
   end + 1wk) so bars/markers aren't all jammed at one edge. Factor the min/max/`pct`
   computation so it's testable.
5. **Keep it a server component where possible** — this is layout/CSS, no new interactivity
   required. Only extract a client piece if an overlay needs measured pixel widths (prefer
   pure CSS percentages to avoid that).

This is a visual fix; preserve the existing color semantics (completed `#588157`, overdue
`#A4503C`, in-progress `#2E4034`, subtask muted greens) and the Forest Floor palette.
