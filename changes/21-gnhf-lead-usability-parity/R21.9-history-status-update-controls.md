# R21.9 — Standing History page was read-only for status updates

**Status:** done (merged in `0b84e64`)
**Files:** `src/app/(app)/projects/[id]/history/page.tsx`, `src/lib/actions/status-updates.ts`

## Spec

**Problem:** Edit/delete controls for status updates existed only on the project detail page's recent-5
list; the prominent "Project Standing History" (see-all) page rendered submissions read-only — the same
cross-surface drift pattern as R21.4.

**Approach:** The history page renders `StatusUpdateControls` using the same `canModifyStatusUpdate` gate as
the project page (MANAGE_STATUS_UPDATES any time, or the submitting lead/sublead before the linked meeting
starts), with `calendarEvent.startsAt` added to its query to compute the edit window.
`updateStatusUpdate`/`deleteStatusUpdate` now also `revalidatePath` the project's `/history` route so edits
from either surface keep both fresh.

## Tests

Retroactive — verified during the run: typecheck + controls appear and function on the history page under
the same gate as the project page. No dedicated Playwright spec.
