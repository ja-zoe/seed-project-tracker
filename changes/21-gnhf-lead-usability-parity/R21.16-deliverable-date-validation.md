# R21.16 — Create paths skipped the deliverable start<=target validation

**Status:** done (merged in `4c45d84`)
**Files:** `src/lib/actions/deliverables.ts`, `src/app/(app)/projects/[id]/deliverables/new/page.tsx`, `src/components/deliverable-date-fields.tsx` (new), `src/app/api/mcp/route.ts`

## Spec

**Problem:** `updateDeliverable`/`updateDeliverableDates` and the edit modal enforce startDate <= targetDate,
but `createDeliverable` and the MCP create/update tools did not — a lead could create an inverted
deliverable. Not cosmetic: the timeline Gantt's `hasBar` requires startDate < targetDate, so an inverted
deliverable silently collapses to a milestone diamond, and the Excel export range is wrong.

**Approach:** `createDeliverable` rejects startDate > targetDate, mirroring the edit paths. New
`DeliverableDateFields` client component gives the create form's Start/Target inputs the same linked
min/max bounds as the edit modal. MCP `create_deliverable` validates start<=target;
`update_deliverable` validates the *effective post-update* window by merging supplied args with the
existing row (fetching startDate/targetDate in the `findUnique`) so a partial update can't invert an
existing deliverable.

## Tests

Retroactive — verified during the run: typecheck + inverted dates rejected on web create, MCP create, and
MCP partial update. No dedicated Playwright spec.
